# 保证金模型：Cross vs Isolated

> 保证金是杠杆交易的基础。这篇深入 Hyperliquid 的两种保证金模式、保证金率计算、以及它们对风险的影响。

---

## 1. 保证金的本质

### 1.1 为什么需要保证金

杠杆交易 = 借钱放大仓位。保证金是你的"抵押品"，保证你有能力承担亏损。

```
10x 杠杆做多 1 BTC（$60,000）：
  你出保证金：$6,000
  系统 "借" 给你：$54,000
  你控制的仓位：$60,000

如果 BTC 跌 → 你的保证金吸收亏损
如果保证金不够吸收 → 清算（防止你欠系统钱）
```

### 1.2 保证金的两个层次

| 类型 | 含义 | 作用 |
|------|------|------|
| **初始保证金（Initial Margin）** | 开仓需要的最低保证金 | 决定最大杠杆 |
| **维持保证金（Maintenance Margin）** | 持仓需要维持的最低保证金 | 决定清算线 |

```
初始保证金率 = 1 / 最大杠杆
  40x → 2.5%
  10x → 10%

维持保证金率 ≈ 初始保证金率 / 2
  40x → 1.25%
  10x → 5%
```

---

## 2. Cross Margin（交叉保证金）

### 2.1 机制

所有持仓**共享一个保证金池**。账户里所有可用余额都是保证金。

```
账户余额：$10,000
持仓 A：BTC 多 10x（保证金占用 $3,000）
持仓 B：ETH 空 5x（保证金占用 $2,000）
可用保证金：$5,000

→ 所有持仓的盈亏汇总到一个账户权益
→ A 的盈利可以支撑 B 的亏损
```

### 2.2 优势

- **资本效率最高**：闲置余额都在做保证金
- **不易单仓爆仓**：整个账户权益都在保护每个仓位
- **盈亏对冲**：一个仓位盈利可以抵消另一个亏损

### 2.3 风险

- **连锁清算**：如果账户权益不足，**所有仓位**可能一起被清算
- **风险传染**：一个仓位的巨亏会拖累所有仓位
- **难以隔离风险**：不能"只赌这一个仓位"

### 2.4 适用场景

- 对冲交易（多空组合）
- 专业交易者（精细管理总风险）
- 资金效率优先的策略

---

## 3. Isolated Margin（逐仓保证金）

### 3.1 机制

每个持仓有**独立的保证金**。一个仓位最多亏完它自己的保证金。

```
账户余额：$10,000
持仓 A：BTC 多 10x（分配 $3,000，isolated）
持仓 B：ETH 空 5x（分配 $2,000，isolated）

→ A 最多亏 $3,000（爆仓后不影响其他）
→ B 最多亏 $2,000
→ 剩余 $5,000 完全安全
```

### 3.2 优势

- **风险隔离**：单仓爆仓不影响其他仓位和余额
- **风险可控**：明确知道每个仓位最多亏多少
- **适合高风险仓位**：赌 meme 币用 isolated，主流仓位用 cross

### 3.3 风险

- **资本效率低**：分配的保证金被锁定
- **更容易被清算**：单仓保证金少，缓冲小
- **不能盈亏对冲**：A 的盈利不能救 B

### 3.4 适用场景

- 高风险投机（meme 币、小币种）
- 想限制单笔最大亏损
- 新手（明确风险边界）

---

## 4. Cross vs Isolated 对比

| 维度 | Cross | Isolated |
|------|-------|----------|
| 保证金池 | 共享 | 独立 |
| 资本效率 | 高 | 低 |
| 风险隔离 | 无 | 有 |
| 盈亏对冲 | 有 | 无 |
| 清算影响 | 可能连锁 | 单仓 |
| 适合 | 对冲、专业 | 投机、新手 |

### 4.1 一个直观的例子

```
场景：你做多 BTC，同时做多一个 meme 币

用 Cross：
  meme 币暴跌 → 拖累账户权益 → 可能连 BTC 仓位一起爆
  
用 Isolated（meme 仓）：
  meme 币暴跌 → 只亏 meme 仓的保证金 → BTC 仓位安全
```

**最佳实践**：主流稳健仓位用 Cross（资本效率），高风险投机用 Isolated（风险隔离）。

## 5. 保证金率与账户健康度

### 5.1 关键公式

```
账户权益 = 账户余额 + 未实现盈亏（所有仓位）

保证金率 = 账户权益 / 持仓名义价值

维持保证金 = 持仓名义价值 × 维持保证金率

健康度 = 账户权益 / 维持保证金
  健康度 > 1：安全
  健康度 = 1：触及清算线
  健康度 < 1：触发清算
```

### 5.2 清算价格计算

```
多头清算价格（近似）：
  清算价 = 开仓价 × (1 - 1/杠杆 + 维持保证金率)

例：10x 做多 BTC @ $60,000，维持保证金率 5%
  清算价 = 60,000 × (1 - 0.1 + 0.05) = 60,000 × 0.95 = $57,000
  → BTC 跌到 ~$57,000 触发清算

空头清算价格（近似）：
  清算价 = 开仓价 × (1 + 1/杠杆 - 维持保证金率)
```

### 5.3 杠杆与清算距离

| 杠杆 | 价格反向波动多少触发清算（近似） |
|------|------------------------------|
| 2x | ~45% |
| 5x | ~18% |
| 10x | ~8.5% |
| 20x | ~4% |
| 40x | ~1.9% |

**杠杆越高，清算线越近**。40x 杠杆下，价格反向波动不到 2% 就爆仓。

## 6. 现货作为保证金

### 6.1 统一账户

Hyperliquid 允许用**现货持仓作为永续保证金**（类似 CEX 的 Portfolio Margin）：

```
你持有 10 ETH 现货（$30,000）
→ 可以用这 10 ETH 作为永续保证金
→ 不需要额外存 USDC
→ 资本效率大幅提升
```

### 6.2 折价（Haircut）

作为保证金的现货可能有折价：
- USDC：100%（无折价）
- 主流币（BTC/ETH）：可能 90-95%
- 波动大的币：折价更多

折价是为了应对现货本身的价格波动风险。

---

## 7. 跨市场保证金的风险

### 7.1 Cross Margin 的系统性风险

回忆 JELLY 事件（[basics/08](../../basics/08-risks-and-events.md)）：
- 攻击者利用保证金机制
- 主动提取保证金触发清算
- 暴露了保证金 + 清算 + HLP 的组合漏洞

### 7.2 保证金设计的安全考量

- 维持保证金率不能太低（否则清算来不及）
- 高波动市场需要更高保证金率
- Cross margin 的连锁风险需要监控

---

## 8. 小结

1. **保证金 = 杠杆交易的抵押品**，吸收亏损，防止欠系统钱
2. **初始保证金**决定最大杠杆，**维持保证金**决定清算线
3. **Cross Margin**：共享池，资本效率高，但可能连锁清算
4. **Isolated Margin**：独立池，风险隔离，但资本效率低
5. **杠杆越高，清算线越近**（40x → 1.9% 波动就爆）
6. **现货可作保证金**（统一账户），但有折价
7. **最佳实践**：稳健仓用 Cross，投机仓用 Isolated
