# Insurance Fund 与 ADL：最后的防线

> 当清算无法在市场上完成时，谁来兜底？这篇拆解 Insurance Fund、ADL（自动减仓）和 HLP 的兜底角色——以及为什么这是 Hyperliquid 最脆弱的环节。

---

## 1. 三层防线

Hyperliquid 的坏账防护有三层：

```
第一层：清算引擎在市场平仓
    │（如果市场流动性不足）
    ▼
第二层：Insurance Fund 吸收损失
    │（如果 Insurance Fund 不足）
    ▼
第三层：ADL（自动减仓）
```

外加一个特殊角色：**HLP Vault** 在某些情况下充当"做市 + 接盘"的缓冲。

---

## 2. Insurance Fund（保险基金）

### 2.1 作用

Insurance Fund 是清算的"安全垫"。当清算产生坏账时，用它来覆盖。

```
正常清算：平仓价好于破产价 → 盈余进 Insurance Fund（基金增长）
坏账清算：平仓价差于破产价 → Insurance Fund 补贴（基金减少）
```

### 2.2 资金来源

| 来源 | 说明 |
|------|------|
| 清算罚金 | 被清算账户的额外费用（0.5-1%） |
| 清算盈余 | 平仓价好于破产价的差额 |
| 平台收入分成 | 部分手续费 |

### 2.3 资金用途

- 覆盖坏账（平仓价差于破产价时）
- 防止 ADL 触发（基金充足时不需要 ADL）
- 作为系统稳定的缓冲

### 2.4 Hyperliquid 的特殊性

Hyperliquid 的 Insurance Fund 和 **HLP Vault** 部分重叠：
- HLP 是 Insurance Fund 的"后备军"
- 在某些情况下 HLP 充当兜底方
- 这个设计在 JELLY 事件中暴露了风险

---

## 3. HLP 的兜底角色

### 3.1 HLP 是什么

HLP（Hyperliquidity Provider）是 Hyperliquid 的官方做市 Vault：
- 用户存 USDC → HLP 自动做市
- 详见 [04-vaults-and-hlp/](../04-vaults-and-hlp/)

### 3.2 HLP 在清算中的角色

HLP 同时是**三个角色**：

```
1. 做市商：在盘口挂单，提供流动性
2. 清算接盘方：当清算无法在市场完成，HLP 接手
3. Insurance Fund 后备：兜底坏账
```

### 3.3 为什么这是风险

```
正常市场：
  清算 → 市场上有对手方 → 正常平仓 → HLP 不需要兜底

低流动性市场：
  清算 → 市场上没对手方 → HLP 被迫接手
  → HLP 持有了一个"没人要的"仓位
  → 如果价格继续不利 → HLP 亏损
```

**HLP 作为中央对手方，在低流动性市场是单点故障**。

---

## 4. JELLY 事件：HLP 兜底的失败案例

### 4.1 事件回顾

（完整版见 [basics/08](../../basics/08-risks-and-events.md)）

```
1. 攻击者在低流动性的 JELLY 永续开巨额空头
2. 在 Solana 现货拉升 JELLY 价格
3. 主动提取保证金 → 触发自己的清算
4. 清算引擎要平空（买入 JELLY perp）→ 市场上没对手方
5. HLP 被迫接手 → 持有巨额 JELLY 多头
6. 攻击者继续拉价 → HLP 浮亏 ~$1000 万
7. 验证者投票下架市场，按危机前价格结算
```

### 4.2 暴露的问题

1. **HLP 是单点故障**：所有低流动性市场的风险都集中到 HLP
2. **Permissionless 上架 + 兜底 = 矛盾**：任何人能上市场，但 HLP 要为所有市场兜底
3. **保证金机制可被利用**：主动提取保证金触发清算是攻击向量

### 4.3 事后改进

| 改进 | 作用 |
|------|------|
| HLP 风险分层 | 不同市场独立计算 HLP 敞口 |
| 单市场敞口上限 | HLP 对单个市场的最大敞口 |
| OI 上限 | 永续 OI 不能超过现货市值一定比例 |
| 断路器 | 价格异常时暂停市场 |

---

## 5. ADL（自动减仓）

### 5.1 什么时候触发

ADL 是**最后手段**：当 Insurance Fund + HLP 都不足以覆盖坏账时。

```
坏账 > Insurance Fund 能覆盖 → 触发 ADL
```

### 5.2 ADL 机制

```
系统找到 "盈利最多 + 杠杆最高" 的对手方
    │
    ▼
强制减少他们的持仓（部分平仓）
    │
    ▼
用这些盈利仓位 "吸收" 坏账
```

### 5.3 ADL 优先级

```
ADL 优先级 = 未实现盈利率 × 有效杠杆

盈利越多 + 杠杆越高 → 越先被 ADL
```

**为什么这么排序**：
- 盈利多的人"赚够了"，被减仓痛感小
- 高杠杆对系统风险贡献大，优先处理
- 比"全员按比例分摊"更公平

### 5.4 ADL 的用户体验

```
你做多 BTC，盈利 50%，20x 杠杆
另一边有人穿仓，Insurance Fund 不够
→ 你可能被 ADL，部分仓位被强制平仓（按标记价）
→ 你收到通知
→ 你的盈利被"锁定"了一部分（不能继续赚）
```

ADL 对被减仓的用户是负面的（打断了盈利仓位），但比"系统坏账拖垮所有人"好。

---

## 6. 三种坏账处理方式对比

| 方式 | 机制 | 谁承担 | 用谁 |
|------|------|--------|------|
| **Insurance Fund** | 基金覆盖 | 基金（来自罚金/盈余） | 理想情况 |
| **ADL** | 减仓盈利对手方 | 盈利最多的用户 | HL、Binance |
| **社会化亏损** | 全员按比例分摊 | 所有盈利用户 | 早期 BitMEX |

Hyperliquid 的顺序：**Insurance Fund → HLP → ADL**。正常情况下 ADL 极少触发。

---

## 7. 系统稳定性分析

### 7.1 正常市场（流动性充足）

```
清算 → 市场平仓 → Insurance Fund 微调 → 系统稳定
ADL 几乎不触发
HLP 正常做市赚钱
```

### 7.2 极端市场（流动性枯竭）

```
清算 → 无对手方 → HLP 接盘 → HLP 亏损
如果 HLP 也撑不住 → ADL
极端情况 → 验证者干预（JELLY 模式）
```

### 7.3 风险的根源

> 永续合约系统的稳定性，本质上取决于"清算时市场有没有对手方"。流动性充足时一切正常，流动性枯竭时所有防线依次承压。

这就是为什么：
- 高流动性市场（BTC/ETH）很安全
- 低流动性市场（meme 币、新上市场）是风险源
- HIP-3 的 permissionless 上架放大了这个风险

---

## 8. 对不同角色的启示

### 8.1 对交易者

- 在低流动性市场，即使你判断对方向，也可能被 ADL
- 高盈利 + 高杠杆的仓位更容易被 ADL
- 主流市场（深流动性）的 ADL 风险极低

### 8.2 对 HLP 存款人

- HLP 的收益来自做市，但风险来自兜底
- JELLY 类事件会让 HLP 亏损
- 不是无风险收益，理解尾部风险

### 8.3 对系统设计者

- 中央对手方（HLP）模式在异类市场不可持续
- 需要"按市场分层的多 Vault 结构"
- Permissionless 上架必须配合严格的风险隔离

---

## 9. 小结

1. **三层防线**：清算引擎 → Insurance Fund → ADL
2. **Insurance Fund**：清算罚金 + 盈余构成，覆盖坏账
3. **HLP 兜底**：HLP 同时是做市商 + 接盘方 + 基金后备（风险点）
4. **JELLY 事件**：HLP 在低流动性市场被迫接盘，浮亏 $1000 万
5. **ADL**：最后手段，减仓"盈利最多 + 杠杆最高"的对手方
6. **稳定性根源**：取决于清算时市场有没有对手方
7. **核心矛盾**：HLP 中央对手方模式 + permissionless 上架 = 持续的尾部风险
