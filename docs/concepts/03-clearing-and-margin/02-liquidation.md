# 清算机制：触发、定价与执行

> 清算是永续合约系统的"安全阀"。这篇深入清算的触发条件、标记价格、清算流程，以及为什么清算设计如此关键。

---

## 1. 清算的本质

### 1.1 为什么需要清算

```
你 10x 做多 1 BTC @ $60,000，保证金 $6,000
BTC 跌到 $54,000 → 你浮亏 $6,000 → 保证金归零

如果不清算，BTC 继续跌到 $53,000：
  → 你浮亏 $7,000
  → 但你只有 $6,000 保证金
  → 你欠系统 $1,000（穿仓）

清算的作用：在你的保证金耗尽前，强制平仓，防止穿仓
```

### 1.2 清算保护谁

- 保护**系统**：防止用户欠债（坏账）
- 保护**对手方**：保证对手方能拿到盈利
- 保护**其他用户**：防止坏账由全体分摊

---

## 2. 清算触发条件

### 2.1 触发逻辑

```
当 账户权益 < 维持保证金 → 触发清算

账户权益 = 余额 + 未实现盈亏
维持保证金 = 持仓名义价值 × 维持保证金率
```

### 2.2 实时监控

清算引擎**持续监控**所有持仓：
- 每个区块更新标记价格
- 重新计算每个账户的健康度
- 健康度 < 1 → 进入清算队列

### 2.3 部分清算 vs 全部清算

Hyperliquid 可能支持**部分清算**：
- 不一定一次平掉全部仓位
- 先平一部分，让账户回到健康水平
- 减少对市场的冲击
- 对用户更友好（保留部分仓位）

---

## 3. 标记价格（Mark Price）

### 3.1 为什么不用最新成交价

```
如果用最新成交价触发清算：
  攻击者用一笔大单瞬间砸盘 → 最新价暴跌
  → 触发大量清算 → 攻击者低价接盘获利
  这叫 "插针清算" 攻击
```

### 3.2 标记价格的计算

```
Mark Price = Oracle Price × (1 + Funding Basis)

Oracle Price = 多个外部交易所的加权中间价
  → 反映全市场共识价格
  → 单个交易所的瞬间波动影响小

Funding Basis = 当前累积的资金费率偏差
```

### 3.3 为什么标记价格更安全

| 价格类型 | 可操纵性 | 用途 |
|---------|---------|------|
| 最新成交价 | 高（一笔单就能动） | 显示 |
| **标记价格** | **低（多交易所加权）** | **清算判断** |
| Oracle 价格 | 中（取决于来源） | 标记价格的输入 |

用标记价格触发清算，攻击者需要操纵**多个交易所**的价格，成本极高。

### 3.4 Oracle 的角色与风险

- Oracle 价格是标记价格的核心输入
- 如果 Oracle 被操纵 → 标记价格错误 → 错误清算
- HIP-3 市场的 Oracle 由 Builder 提供 → 是潜在风险点
- 主流市场用多源 Oracle（Pyth、Chainlink 等）→ 更安全

## 4. 清算流程

### 4.1 完整流程

```
1. 标记价格更新 → 账户健康度 < 1
    │
    ▼
2. 清算引擎接管该账户持仓
    │
    ▼
3. 计算破产价格（Bankruptcy Price）
   = 保证金完全耗尽的价格
    │
    ▼
4. 尝试在市场上平仓（限价单方式）
    │
    ├─ 市场流动性充足 → 正常平仓
    │   ├─ 平仓价好于破产价 → 盈余进 Insurance Fund
    │   └─ 平仓价差于破产价 → Insurance Fund 补贴
    │
    └─ 市场流动性不足 → HLP 接手 / ADL（见下一篇）
    │
    ▼
5. 清算罚金 → Insurance Fund
```

### 4.2 破产价格 vs 清算价格

```
清算价格：触发清算的价格（账户权益 = 维持保证金）
破产价格：保证金完全耗尽的价格（账户权益 = 0）

清算价格 和 破产价格 之间有一个缓冲区
  → 这个缓冲让清算引擎有空间平仓
  → 理想情况下在缓冲区内平完，不产生坏账
```

### 4.3 清算罚金

- 被清算时收取额外费用（~0.5-1%）
- 罚金进入 Insurance Fund
- 作用：补偿系统风险 + 抑制过度杠杆

## 5. 清算的市场影响

### 5.1 清算级联（Liquidation Cascade）

```
价格下跌 → 触发一批多头清算 → 清算引擎卖出平仓
  → 卖压推动价格进一步下跌
  → 触发更多清算
  → ...（级联）
```

这是加密市场剧烈波动的常见原因。大量同向高杠杆仓位 + 价格波动 = 清算级联。

### 5.2 Hyperliquid 的缓解机制

| 机制 | 作用 |
|------|------|
| 标记价格 | 防止插针触发 |
| 部分清算 | 减少单次冲击 |
| 断路器 | 极端波动时暂停市场 |
| 杠杆上限 | 限制最大杠杆（40x），减少级联 |
| OI 上限 | 限制单市场总持仓 |

### 5.3 和 CEX 的对比

```
CEX 清算：
  不透明（你不知道有多少仓位接近清算）
  
Hyperliquid 清算：
  链上透明（清算价格、清算量都可查）
  → 可以监控市场的清算风险
  → 但也意味着攻击者也能看到（双刃剑）
```

## 6. 用户如何避免清算

### 6.1 实操建议

| 策略 | 说明 |
|------|------|
| **降低杠杆** | 杠杆越低，清算线越远 |
| **设置止损** | 主动平仓，好于被清算（无罚金） |
| **监控保证金率** | 接近清算线时加保证金 |
| **用 Isolated** | 隔离高风险仓位 |
| **避免高波动时机** | 重大事件前减仓 |

### 6.2 止损 vs 清算

```
主动止损：
  你设定的价格平仓
  只付正常手续费
  保留剩余保证金

被动清算：
  系统强制平仓
  付清算罚金（0.5-1%）
  可能在更差的价格成交

→ 永远优先用止损，不要等清算
```

## 7. 清算设计的核心权衡

### 7.1 速度 vs 冲击

```
清算太快/太激进：
  → 对市场冲击大（清算级联）
  → 用户损失大

清算太慢/太温和：
  → 可能来不及（穿仓坏账）
  → 系统风险
```

Hyperliquid 用"部分清算 + 标记价格 + 缓冲区"来平衡。

### 7.2 低流动性市场的困境

JELLY 事件暴露的核心问题：

```
低流动性市场：
  清算引擎要平仓 → 但市场上没有对手方
  → 无法在合理价格平仓
  → HLP 被迫接手（在扭曲的价格）
  → HLP 亏损
```

这是清算机制在低流动性市场的根本困境。详见下一篇的 HLP 和 ADL。

---

## 8. 小结

1. **清算防止穿仓**：在保证金耗尽前强制平仓
2. **触发条件**：账户权益 < 维持保证金
3. **标记价格**：多交易所加权，防止插针清算
4. **清算流程**：接管 → 计算破产价 → 市场平仓 → 盈余/补贴
5. **清算级联**是市场剧烈波动的常见原因
6. **缓解机制**：标记价格、部分清算、断路器、杠杆/OI 上限
7. **用户应对**：降杠杆、设止损（永远好于被清算）
8. **核心困境**：低流动性市场清算无对手方（JELLY 事件根源）
