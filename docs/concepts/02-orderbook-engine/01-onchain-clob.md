# 全链上 CLOB：DeFi 五年的圣杯

> "在链上跑一个像样的订单簿"是 2020 年以来 DeFi 最难的工程挑战。这篇解释为什么它这么难，以及 Hyperliquid 怎么做到的。

---

## 1. 什么是 CLOB

### 1.1 中央限价订单簿

**CLOB（Central Limit Order Book，中央限价订单簿）**是传统金融用了一百年的交易机制。

```
卖单（Asks）          
$60,200 | 5 BTC    ← 卖方挂单
$60,150 | 3 BTC
$60,100 | 2 BTC    ← 最优卖价（best ask）
─────────────────  ← 价差（spread）
$60,050 | 4 BTC    ← 最优买价（best bid）
$60,000 | 6 BTC
$59,950 | 8 BTC    ← 买方挂单
买单（Bids）
```

**撮合规则**：价格优先 → 时间优先
- 买价 ≥ 卖价 → 成交
- 同价格，先挂的先成交

### 1.2 CLOB vs AMM

| 维度 | CLOB | AMM |
|------|------|-----|
| 定价 | 市场博弈 | 数学公式（x*y=k） |
| 做市 | 需要做市商 | 任何人提供 LP |
| 价格发现 | 有 | 无（跟随套利） |
| 资本效率 | 高 | 低 |
| 滑点 | 取决于深度 | 取决于池子大小 |
| 链上难度 | **极高** | 低 |

**关键**：CEX 全是 CLOB，DEX 过去大部分是 AMM。原因就是 CLOB 上链太难。

## 2. 为什么链上 CLOB 这么难

### 2.1 订单簿的高频特性

一个活跃的订单簿，每秒钟发生：
- 成千上万次下单
- 成千上万次撤单
- 成千上万次修改

**做市商的行为模式**：
- 持续报价（挂买单和卖单）
- 价格变化时**立即撤单重挂**
- 一个做市商每秒可能下/撤几百到几千个订单

### 2.2 Gas 成本的致命问题

在 EVM 链上，每个状态变化都要付 Gas：

```
做市商每秒撤/挂 1000 个订单
每个订单上链 Gas ~$0.01-1（取决于链）
每秒成本：$10-1000
每天成本：$864,000 - $86,400,000
```

**这是不可承受的**。没有做市商会在这种成本下报价。没有做市商 → 没有流动性 → 订单簿是死的。

### 2.3 出块时间的问题

```
以太坊出块 12 秒：
  你挂单 → 12 秒后才上链
  价格早就变了
  → 你的报价是"过时的"

做市需要毫秒级的挂撤单
12 秒出块根本没法做市
```

### 2.4 Serum 的失败（活的反例）

Serum 是 Solana 上的链上 CLOB 尝试（2021-2022）。它的失败教训：

| 问题 | 原因 |
|------|------|
| 撮合是合约 | 撮合逻辑跑在 Solana 程序里，和其他应用抢 Compute Unit |
| 负载下卡顿 | NFT mint 或热门 launch 时 Serum 卡住 |
| 做市商不来 | 挂单要付 Gas，专业做市商不愿意 |
| 依赖 FTX | FTX 倒塌后前端和基础设施全挂 |

**核心教训**：即使在 Solana（最快的通用 VM）上，把撮合做成**合约**仍然不够。

## 3. Hyperliquid 的解法

### 3.1 核心洞察

> 前人都在问"怎么把 CLOB 塞进现有 VM"。
> Hyperliquid 问的是"**如果链本身就是为订单簿设计的呢**？"

### 3.2 撮合是原生状态机，不是合约

**这是最关键的区别**：

```
Serum / 其他尝试：
  订单簿 = 智能合约（跑在通用 VM 上）
  → 和其他应用抢资源
  → 受 VM 的 Gas 模型限制

Hyperliquid：
  订单簿 = L1 的原生状态机（是共识的一部分）
  → 不和任何应用抢资源
  → 不受 EVM Gas 模型限制
```

**HyperCore 维护的原生状态**：
- 每个市场的订单簿（买卖盘）
- 每个账户的余额、持仓、保证金
- 资金费率、清算状态

这些都是**原生数据结构**，不是合约里的 storage。操作它们是**原生 opcode**，不烧 EVM Gas。

### 3.3 类比：专用芯片 vs 通用 CPU

```
通用 VM 上的 CLOB（Serum）
= 用通用 CPU 跑图形渲染
= 能跑，但慢、贵、抢资源

原生状态机的 CLOB（Hyperliquid）
= 用专用 GPU 跑图形渲染
= 为这个任务定制，快、专用
```

Hyperliquid 本质上是为"订单簿"这个任务造了一个"专用芯片"（一条链）。

## 4. 全链上的含义

### 4.1 "链上"的三个层次

回忆 [basics 里提到的](../../web3-insider) 三个层次：

| 层次 | 含义 | 例子 |
|------|------|------|
| 链上结算 | 撮合在链下，结算在链上 | dYdX V3（StarkEx） |
| 链上撮合 | 撮合是合约 | Serum |
| **链上 + 原生** | 撮合是 L1 共识的一部分 | **Hyperliquid** |

### 4.2 Hyperliquid 是第三种

- 每个订单都是 L1 交易（被验证者共识）
- 撮合逻辑是 L1 状态机（不是合约）
- 完全可验证（理论上，但代码不开源）

### 4.3 全链上的好处

- **可验证**：每个订单、每次撮合都在链上（理论上可审计）
- **自托管**：用户资金不离开自己的控制
- **无信任**：不需要信任某个中心化撮合方
- **透明**：盘口、成交、持仓都公开

### 4.4 全链上的代价

- **对链的性能要求极高**（所以需要 HyperBFT 的 200K TPS）
- **状态膨胀**（订单簿状态大）
- **必须自建链**（通用链做不到）

## 5. 和 dYdX V4 的对比

dYdX V4 也是"去中心化订单簿"，但实现不同：

| 维度 | dYdX V4 | Hyperliquid |
|------|---------|-------------|
| 订单簿位置 | 验证者**内存**中 | L1 **原生状态** |
| 撮合 | 链下（内存）撮合 | 链上原生撮合 |
| 上链内容 | 只有成交结果 | 订单 + 成交都在链上 |
| 可验证性 | 部分（只能验证成交） | 完全（理论上） |
| 性能 | 好 | 更好 |

**关键区别**：
- dYdX V4 的订单簿在验证者内存里，**只有成交才写链**
- Hyperliquid 的订单簿**本身就是链上状态**

dYdX 的方式验证者负担小，但牺牲了部分可验证性。Hyperliquid 的方式更"纯粹"，但对链性能要求更高。

---

## 6. 小结

1. **CLOB 是金融的标准机制**，但上链极难（高频 + Gas + 出块时间）
2. **Serum 证明了"撮合做成合约"不够**（即使在 Solana 上）
3. **Hyperliquid 的解法：撮合是 L1 原生状态机**，不是合约
4. **类比专用芯片**：为订单簿造一条链，而非在通用链上做订单簿
5. **全链上 + 原生**是最纯粹的形态，代价是必须自建高性能链
6. **vs dYdX V4**：dYdX 订单簿在内存，Hyperliquid 在链上状态
