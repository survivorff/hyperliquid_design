# 订单生命周期：从提交到成交的完整路径

> 一笔订单在 Hyperliquid 上经历了什么？这篇追踪一笔订单从用户点击到最终成交的每一步，以及订单的各种状态。

---

## 1. 完整数据流

### 1.1 端到端路径

```
用户（浏览器/API）
    │
    │ 1. 构造订单对象
    │ 2. EIP-712 签名
    ↓
验证者节点
    │
    │ 3. 验证签名
    │ 4. 检查余额/保证金
    │ 5. 不合法 → 拒绝（订单不进入区块）
    ↓
HyperBFT 共识
    │
    │ 6. 订单进入共识，确定顺序（~100-200ms）
    ↓
HyperCore 执行
    │
    │ 7. 订单进入订单簿
    │ 8. 撮合检查
    │    ├─ 有匹配 → 成交 → 更新余额/持仓
    │    └─ 无匹配 → 挂单等待
    ↓
状态确认
    │
    │ 9. 最终性达成（0.2s）
    │ 10. WebSocket 推送结果
    ↓
用户看到结果
```

### 1.2 时间拆解

| 阶段 | 耗时 | 说明 |
|------|------|------|
| 用户 → 验证者 | ~50ms | 网络延迟 |
| 验证 | ~10ms | 签名 + 余额检查 |
| 共识 | ~100-150ms | HyperBFT 投票 |
| 撮合 | ~1-5ms | 订单簿匹配 |
| 推送 | ~50ms | WebSocket 回传 |
| **总计** | **~200-300ms** | 端到端 |

## 2. 订单状态机

### 2.1 状态流转

```
[提交]
   │
   ▼
PENDING（待处理）
   │
   ├─ 验证失败 → REJECTED（拒绝）[终态]
   │
   ▼
（进入共识 + 订单簿）
   │
   ├─ 立即完全成交 → FILLED（完全成交）[终态]
   │
   ├─ 立即部分成交 → PARTIALLY_FILLED（部分成交）
   │                      │
   │                      ▼
   │                  剩余挂在簿上 → OPEN
   │
   └─ 无匹配 → OPEN（挂单中）
                  │
                  ├─ 对手方出现 → 成交 → FILLED
                  ├─ 用户撤单 → CANCELED（已撤销）[终态]
                  └─ 触发条件（止损等）→ 激活 → 撮合
```

### 2.2 状态说明

| 状态 | 含义 | 是否终态 |
|------|------|:---:|
| PENDING | 已提交，待验证/共识 | 否 |
| REJECTED | 验证失败（余额/保证金不足） | 是 |
| OPEN | 挂在订单簿上等待 | 否 |
| PARTIALLY_FILLED | 部分成交，剩余继续挂单 | 否 |
| FILLED | 完全成交 | 是 |
| CANCELED | 用户主动撤销 | 是 |

### 2.3 和 EVM DEX 的关键区别

```
EVM DEX：
  订单可能 "上链后失败（revert）"
  → 有 "FAILED" 状态（付了 Gas 但没成交）

Hyperliquid：
  不合法的订单在进入区块前就被 REJECTED
  → 进入区块的订单不会失败
  → 没有 "上链后失败" 这个状态
```

这是确定性执行的体现（详见 [01-hyperbft-consensus/04](../01-hyperbft-consensus/04-finality-and-determinism.md)）。

## 3. 订单的构造与签名

### 3.1 订单对象

一个订单包含：

| 字段 | 说明 |
|------|------|
| asset | 交易对（如 BTC-PERP） |
| isBuy | 方向（买/卖） |
| size | 数量 |
| limitPrice | 限价（市价单为可接受的最差价） |
| orderType | 限价/市价/止损等 |
| reduceOnly | 是否只减仓 |
| timeInForce | GTC/IOC/FOK/ALO |
| cloid | 客户端订单 ID（可选） |

### 3.2 EIP-712 签名

订单通过 **EIP-712** 结构化签名：
- 用户用钱包私钥签名订单
- 签名证明"这个订单确实是我发的"
- 验证者验证签名，确认订单合法性
- **不需要把私钥暴露给任何人**（自托管）

### 3.3 签名的意义

```
中心化交易所：
  你登录 → 服务器信任你的 session → 你下单

Hyperliquid：
  你签名每个订单 → 验证者验证签名 → 自托管
  → 不需要 "把资金托管给交易所"
```

## 4. 挂单与撤单

### 4.1 挂单（Maker）

如果订单没有立即成交，它会挂在订单簿上：
- 状态变为 OPEN
- 成为流动性提供方（Maker）
- 享受 Maker 费率（0.01%）
- 等待对手方出现

### 4.2 撤单

```
用户发起撤单
    │
    ▼
验证者验证（这个订单是你的吗？）
    │
    ▼
共识确认顺序
    │
    ▼
从订单簿移除
    │
    ▼
状态变为 CANCELED
```

**撤单是 Gas-free 的**（见 [02-matching-and-gasfree.md](./02-matching-and-gasfree.md)）。

### 4.3 撤单的竞态

```
你想撤单，但对手方同时来成交：
  情况 A：撤单先进共识 → 撤单成功，订单 CANCELED
  情况 B：成交先进共识 → 订单已 FILLED，撤单失败（订单已不存在）

因为顺序是确定的，结果是确定的（不存在 "既撤了又成交了" 的矛盾）
```

## 5. 条件单的生命周期

### 5.1 止损/止盈单

条件单的特殊之处：**提交时不进订单簿，等触发条件满足才激活**。

```
提交止损单（触发价 $58,000，做空止损）
    │
    ▼
状态：OPEN（但不在活跃订单簿，是 "待触发" 状态）
    │
    ▼
标记价格跌到 $58,000
    │
    ▼
触发 → 转为市价单/限价单 → 进入撮合
    │
    ▼
成交 → FILLED
```

### 5.2 触发基于标记价格

条件单的触发基于**标记价格（Mark Price）**而非最新成交价：
- 防止 "插针" 触发（瞬间价格操纵）
- 见 [basics/04-trading-mechanism.md](../../basics/04-trading-mechanism.md) 第 5.2 节

## 6. TWAP 订单的生命周期

### 6.1 TWAP 的拆分执行

```
提交 TWAP（总量 100 BTC，时间窗口 30 分钟）
    │
    ▼
系统拆成多个子单（如每 30 秒一个 ~1.67 BTC）
    │
    ▼
每个子单按时间点依次提交、撮合
    │
    ├─ 子单 1 → 成交
    ├─ 子单 2 → 成交
    ├─ ...
    └─ 子单 60 → 成交
    │
    ▼
全部完成 → TWAP FILLED
```

### 6.2 TWAP 的中途状态

- 可以中途取消（剩余子单不再执行）
- 已执行部分保留
- 适合大单减少市场冲击

## 8. 小结

1. **端到端 ~200-300ms**：提交 → 验证 → 共识 → 撮合 → 推送
2. **状态机**：PENDING → (REJECTED / OPEN / PARTIALLY_FILLED / FILLED / CANCELED)
3. **无 "上链后失败"**：不合法订单在进区块前就被拒绝（确定性执行）
4. **EIP-712 签名**：每个订单签名，自托管，无需信任托管方
5. **Gas-free 挂撤单**：挂单成为 Maker，撤单免费
6. **条件单**：基于标记价格触发，防插针
7. **TWAP**：拆成子单依次执行，减少市场冲击
