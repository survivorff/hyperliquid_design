# 撮合引擎与 Gas-free 订单

> 为什么在 Hyperliquid 上下单和撤单不要钱？这篇拆解撮合引擎的工作原理和 Gas-free 设计的实现机制。

---

## 1. 撮合引擎

### 1.1 撮合规则：价格优先，时间优先

和 Nasdaq、NYSE、Binance 完全一样的规则：

```
价格优先：
  买单：出价越高，越优先成交
  卖单：要价越低，越优先成交

时间优先（同价格）：
  先挂的订单先成交（FIFO）
```

### 1.2 撮合的触发

```
新订单进入 → 检查对手盘
    │
    ├─ 限价买单 $60,100，最优卖价 $60,050
    │   → 60,100 ≥ 60,050 → 成交（以 $60,050 成交）
    │
    ├─ 限价买单 $60,000，最优卖价 $60,050
    │   → 60,000 < 60,050 → 不成交 → 挂在买盘
    │
    └─ 市价买单
        → 立即和最优卖价成交，吃到足够数量为止
```

### 1.3 撮合的价格改善

如果你的限价买单价格高于最优卖价，**以卖价成交**（价格改善）：

```
你挂限价买 $60,100
最优卖价 $60,050
→ 成交价 = $60,050（对你更有利）
→ 你"省"了 $50
```

这是 CLOB 相对 AMM 的优势之一：你可能以更好的价格成交。

### 1.4 撮合流程（完整）

```
1. 用户提交订单（签名）
2. 验证者验证（余额、保证金、签名）—— 不合法直接拒绝
3. 订单进入 HyperBFT 共识（确定顺序）
4. 订单进入 HyperCore 订单簿
5. 撮合引擎检查对手盘
   ├─ 有匹配 → 成交 → 更新双方余额/持仓
   └─ 无匹配 → 挂在簿上等待
6. 成交结果写入链上状态
7. WebSocket 推送给用户
```

## 2. Gas-free 订单：核心创新

### 2.1 现象

在 Hyperliquid 上：
- **下单不要钱**
- **撤单不要钱**
- **修改订单不要钱**
- **只有成交时**才涉及状态变化（但也不收传统 Gas）

这让做市商可以每秒下/撤几千个订单而零成本。

### 2.2 为什么 EVM 链做不到

在 EVM 链上，每个状态变化都要付 Gas：

```
下单 = 写一个 storage slot（订单数据）= 付 Gas
撤单 = 删一个 storage slot = 付 Gas（虽然有退款）
修改 = 改 storage = 付 Gas
```

每个操作都改变持久状态，所以都要付费。

### 2.3 Hyperliquid 的实现原理

**关键洞察：下单/撤单不应该改变"持久状态"**。

```
传统理解：
  下单 → 订单成为链上持久状态 → 付 Gas

Hyperliquid：
  下单 → 订单进入"订单簿"（一种特殊的、高频变化的状态）
  撤单 → 从订单簿移除
  → 这些操作被特殊处理，不按传统 Gas 计费
```

**推测的机制**：
- 订单簿是 HyperCore 的原生数据结构（不是 EVM storage）
- 下单/撤单是原生操作（不是 EVM opcode）
- 验证者处理这些操作的成本被"摊销"到协议层（用交易手续费覆盖）
- **只有成交**才真正改变账户余额/持仓（这部分成本由成交手续费覆盖）

### 2.4 为什么这样设计是合理的

```
经济逻辑：
  下单/撤单本身不产生价值转移（只是"意图"）
  成交才产生价值转移（钱和仓位易手）

所以：
  对"意图"收费（下单 Gas）= 不合理
  对"成交"收费（手续费）= 合理
```

Hyperliquid 把成本从"每次操作"转移到了"实际成交"。做市商挂大量单但只有少数成交 → 成本低。

## 3. Gas-free 对流动性的影响

### 3.1 做市商友好

```
EVM 链上做市：
  挂 1000 个单/秒 × Gas $0.01 = $10/秒成本
  → 做市商不来 → 流动性差

Hyperliquid 做市：
  挂 1000 个单/秒 × $0 = $0 成本
  → 做市商踊跃 → 流动性深
```

### 3.2 流动性深度的根本原因

Hyperliquid 流动性深的原因链：

```
Gas-free 订单
    ↓
做市商可以高频报价、紧密报价
    ↓
盘口价差小、深度大
    ↓
用户成交体验好
    ↓
更多用户来交易
    ↓
更多做市商来赚价差
    ↓
（正向飞轮）
```

### 3.3 实测数据

| 平台 | BTC 盘口深度（$1M 滑点） | Gas 模型 |
|------|------------------------|---------|
| **Hyperliquid** | **~0.01-0.02%** | 订单免费 |
| dYdX V4 | ~0.03-0.05% | 订单免费 |
| GMX V2 | N/A（Oracle 定价） | 每笔付 Gas |
| Solana DEX | ~0.05-0.1% | 每笔付 ~$0.001 |
| Binance | ~0.005-0.01% | 无 Gas |

Hyperliquid 是链上流动性最深的，接近 Binance。

## 4. 防滥用机制

### 4.1 免费订单的滥用风险

如果下单完全免费，会不会有人恶意刷单（DoS 攻击）？

```
攻击者：每秒发 100 万个垃圾订单
→ 撑爆订单簿
→ 拖慢系统
```

### 4.2 Hyperliquid 的防护（推测）

| 机制 | 作用 |
|------|------|
| **频率限制（Rate Limit）** | 每个地址每秒订单数上限 |
| **保证金要求** | 下单需要足够保证金（垃圾单也要锁保证金） |
| **取消比例限制** | 下单/成交比例过高可能被限制 |
| **账户价值门槛** | 低价值账户的订单频率受限 |

**核心思路**：免费但不是无限制。用"必须有真实保证金 + 频率限制"防止滥用。

### 4.3 和 CEX 的相似性

CEX 也面临同样问题，解法类似：
- API 频率限制
- 订单/成交比例监控
- 做市商有更高额度（因为他们提供流动性）

## 5. 撮合的确定性

### 5.1 撮合结果可预测

因为：
- 订单顺序确定（确定性排序）
- 撮合规则固定（价格-时间优先）
- 所以撮合结果在订单进入时就确定

### 5.2 无 MEV 的撮合

```
EVM DEX：
  你的大单进 mempool → 抢跑者插队 → 你被三明治

Hyperliquid：
  你的订单按确定顺序撮合 → 无法插队 → 无三明治
```

详见 [01-hyperbft-consensus/04-finality-and-determinism.md](../01-hyperbft-consensus/04-finality-and-determinism.md)。

---

## 6. 小结

1. **撮合规则**：价格优先 + 时间优先（和 CEX 一样）
2. **价格改善**：限价单可能以更优的对手价成交
3. **Gas-free 的核心**：下单/撤单不改变"持久状态"，只有成交才计费
4. **经济逻辑**：对"意图"免费，对"成交"收费
5. **流动性飞轮**：Gas-free → 做市商踊跃 → 深度好 → 用户多 → 更多做市商
6. **防滥用**：免费但有频率限制 + 保证金要求
7. **确定性撮合**：结果可预测，无 MEV
