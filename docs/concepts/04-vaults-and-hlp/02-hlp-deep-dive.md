# HLP 深度拆解：系统的"中央对手方"

> HLP 是 Hyperliquid 最核心也最被误解的组件。它既是收益机器，也是系统风险的集中点。这篇彻底拆解它。

---

## 1. HLP 是什么

### 1.1 定义

**HLP（Hyperliquidity Provider）**是 Hyperliquid 官方运营的做市 Vault：
- 用户存入 USDC
- HLP 用这些资金在全市场做市
- 收益分配给存款人
- 是平台流动性的重要来源

### 1.2 规模与地位

| 指标 | 值（估计） |
|------|-----------|
| 规模 | 数亿美元 |
| 历史年化 | 10-30%（波动大） |
| 角色 | 做市 + 清算接盘 + 基金后备 |
| 重要性 | 平台流动性的核心支柱之一 |

---

## 2. HLP 的三重角色

这是理解 HLP 的关键：它同时扮演三个角色，这既是它的价值，也是它的风险。

### 2.1 角色一：主动做市商

```
HLP 在每个市场的盘口挂买单和卖单
赚取买卖价差（spread）
为平台提供基础流动性
```

- 类似传统做市商
- 但用的是存款人的钱
- 收益来自价差

### 2.2 角色二：清算接盘方

```
当清算无法在市场上完成（没有对手方）
HLP 按规则接手被清算的仓位
```

- 在正常市场，这是有利可图的（接手价好于市场价）
- 在低流动性市场，这是亏损的（JELLY 事件）

### 2.3 角色三：Insurance Fund 后备

```
当 Insurance Fund 不足以覆盖坏账
HLP 充当后备资金
```

- 系统级的最后防线之一
- 把系统风险转移到 HLP 存款人

### 2.4 三重角色的张力

```
做市（赚钱）+ 清算接盘（可能亏）+ 基金后备（兜底）
= HLP 存款人既享受做市收益，也承担系统尾部风险
```

**这是 HLP 的本质矛盾**：收益是日常的（做市价差），风险是尾部的（极端事件兜底）。

---

## 3. HLP 的收益来源

### 3.1 收益构成

| 来源 | 说明 | 稳定性 |
|------|------|--------|
| **做市价差** | 买卖盘口的价差 | 稳定 |
| **清算盈利** | 接手清算仓位的盈利（接手价好于市场） | 中等 |
| **手续费分成** | 平台 taker fee 的一部分 | 稳定 |
| **资金费率** | 持有仓位时收/付资金费 | 波动 |

### 3.2 历史收益

```
牛市/正常市场：年化 15-30%
震荡市场：年化 10-15%
极端事件月份：可能为负（JELLY 时浮亏）
```

### 3.3 为什么收益看起来"很香"

```
做市是个 "正期望" 生意：
  大多数时候赚价差
  偶尔在极端行情亏

所以 HLP 长期看是赚钱的
但 "偶尔的亏" 可能很大（尾部风险）
```

---

## 4. HLP 的风险

### 4.1 单边敞口风险

```
极端单边行情（比如 BTC 暴跌）：
  所有人都在卖 → HLP 作为做市商被迫接盘（买入）
  → HLP 累积大量多头
  → 价格继续跌 → HLP 亏损
```

做市商在趋势行情中天然处于不利位置（逆势接盘）。

### 4.2 低流动性市场风险（JELLY 类）

```
低流动性市场清算 → 没对手方 → HLP 被迫接手扭曲价格的仓位
→ 这是 HLP 最大的风险源
```

详见 [03-clearing-and-margin/03](../03-clearing-and-margin/03-insurance-fund-and-adl.md)。

### 4.3 系统性风险

```
多个市场同时出问题：
  HLP 同时在多个市场承压
  → 资金可能不足
  → 触发 ADL 或验证者干预
```

### 4.4 Oracle 操纵风险

```
如果某市场 Oracle 被操纵：
  HLP 可能在错误价格做市/接盘
  → 亏损
```

---

## 5. JELLY 事件中的 HLP

### 5.1 HLP 如何被攻击

```
1. JELLY 是低流动性市场，HLP 是主要做市方
2. 攻击者开巨额空头 + 拉升现货
3. 主动触发清算 → 清算引擎要买入平空
4. 市场没对手方 → HLP 被迫接手（持有 JELLY 多头）
5. 攻击者继续拉价 → HLP 多头浮亏扩大
6. HLP 浮亏 ~$1000 万（总资本的 10-15%）
```

### 5.2 为什么 HLP 是单点故障

```
所有低流动性市场的清算风险
    ↓
都集中到一个 HLP
    ↓
任何一个市场被攻击 → HLP 承压
    ↓
HLP 是系统的 "单点故障"
```

### 5.3 事后的 HLP 改进

| 改进 | 作用 |
|------|------|
| 风险分层 | 不同市场独立计算 HLP 敞口 |
| 单市场敞口上限 | HLP 对单个市场最大敞口限制 |
| 异类市场隔离 | meme 币等和主流市场分开 |

**但根本问题未解**：HLP 作为中央对手方，每个新市场都是新的尾部风险。

## 6. HLP 的设计哲学

### 6.1 为什么 Hyperliquid 要做 HLP

```
冷启动问题：
  新交易所没有做市商 → 没流动性 → 没用户
  
HLP 的解法：
  平台自己做市（用户存款）→ 提供初始流动性
  → 吸引用户 → 吸引专业做市商
```

HLP 是 Hyperliquid 解决"流动性冷启动"的关键。

### 6.2 HLP vs 纯第三方做市

```
纯第三方做市：
  平台不承担做市风险
  但冷启动难（没人来做市）

HLP 模式：
  平台用用户资金做市
  快速建立流动性
  但把做市风险转移给 HLP 存款人
```

### 6.3 中心化的争议

```
HLP 是 "官方" Vault：
  策略由团队决定
  接盘规则由团队设定
  → 这是一种中心化

但：
  存款是自愿的
  收益和风险透明
  → 用户知情同意
```

## 7. 对 HLP 存款人的建议

### 7.1 理解你在承担什么

```
你存 HLP，本质上是：
  做多 "Hyperliquid 做市生意的长期盈利"
  做空 "极端尾部事件"

= 大多数时候赚钱，偶尔大亏
```

### 7.2 实操建议

| 建议 | 理由 |
|------|------|
| 不要 all-in | 尾部风险真实存在 |
| 理解历史回撤 | JELLY 时 HLP 浮亏 10-15% |
| 长期持有 | 做市是长期正期望 |
| 关注风险参数变化 | 事后改进会影响收益/风险 |
| 分散 | 别把所有钱放一个 Vault |

### 7.3 HLP 适合谁

- 想要被动收益、能承受波动的用户
- 理解"这不是无风险存款"的用户
- 长期视角的用户

**不适合**：
- 把它当"高息存款"的用户
- 不能承受 10-20% 回撤的用户

---

## 8. 小结

1. **HLP 是官方做市 Vault**，平台流动性的核心支柱
2. **三重角色**：做市商 + 清算接盘方 + Insurance Fund 后备
3. **收益**：做市价差 + 清算盈利 + 手续费分成（长期正期望）
4. **风险**：单边敞口 + 低流动性接盘 + 系统性 + Oracle（尾部）
5. **JELLY 事件**：HLP 作为中央对手方在低流动性市场的单点故障暴露
6. **设计目的**：解决流动性冷启动问题
7. **存款人本质**：做多做市盈利 + 做空尾部事件
8. **核心矛盾**：日常赚钱，极端大亏；中央对手方模式难持续
