# HyperBFT 的设计：解耦、流水线、乐观执行

> Hotstuff 是基础，但 HyperBFT 在它之上做了为交易场景定制的优化。这篇拆解这些优化的具体机制和工程权衡。

---

## 1. HyperBFT 的设计目标

HyperBFT 不是通用区块链共识，它是**为交易所定制**的共识。设计目标的优先级：

```
1. 低延迟（亚秒最终性）        ← 最高优先级
2. 高吞吐（每秒几万笔订单）
3. 确定性（无 MEV、无 revert）
4. 去中心化                    ← 最低优先级（暂时妥协）
```

**这个优先级排序解释了 HyperBFT 的所有设计决策**。当延迟和去中心化冲突时，HyperBFT 选延迟。

> ⚠️ HyperBFT 代码不开源。本文基于官方文档、白皮书描述和 Hotstuff 家族的通用原理推测，推测处会标注。

---

## 2. 核心优化一：共识与执行解耦

### 2.1 什么是解耦

传统区块链：共识和执行是**串行**的。

```
传统（串行）：
  收到交易 → 执行交易 → 把结果打包 → 共识 → 确认
  （执行必须在共识之前或之中完成）
```

HyperBFT：共识和执行**解耦**，在不同线程/阶段进行。

```
HyperBFT（解耦）：
  共识线程：只对"交易顺序"达成一致（不关心执行结果）
  执行线程：拿到确定的顺序后，独立执行
```

### 2.2 为什么解耦能提速

**关键洞察：达成"顺序共识"比达成"状态共识"快得多**。

- 顺序共识：只需要同意"交易 A 在交易 B 之前"
- 状态共识：需要同意"执行后账户余额是多少"

解耦后：
- 共识层只排序，不执行 → 共识更轻、更快
- 执行层拿到确定顺序后执行 → 因为顺序确定，执行结果也确定
- 两者可以流水线并行

### 2.3 这和 Hyperliquid 架构的关系

回忆 [basics/03](../../basics/03-technical-architecture.md) 里的三层架构：
- HyperBFT（共识）只负责排序
- HyperCore（执行）拿到顺序后撮合
- 这正是"解耦"思想的体现

## 3. 核心优化二：流水线化

### 3.1 Hotstuff 流水线的继承

HyperBFT 继承了 Hotstuff 的流水线设计（见 [01-bft-background](./01-bft-background.md) 第 4.4 节）：

```
区块 N：    Propose → Vote → Commit
区块 N+1：          Propose → Vote → Commit
区块 N+2：                  Propose → Vote → Commit
```

每个阶段的 QC（Quorum Certificate）同时推进多个区块。

### 3.2 HyperBFT 的流水线强化

**推测的优化**（基于性能数据反推）：
- 更激进的流水线深度（更多区块同时在途）
- 更短的区块间隔（~0.1-0.2 秒）
- 投票聚合优化（门限签名的快速验证）

**结果**：
- 理论吞吐 ~200 万 orders/秒
- 实际吞吐 ~20 万 orders/秒
- 出块间隔 ~0.1-0.2 秒

## 4. 核心优化三：乐观执行

### 4.1 什么是乐观执行

**乐观执行（Optimistic Execution）**：在共识完全达成前，就开始执行交易。

```
保守执行：
  等共识 100% 确认 → 才执行
  （安全但慢）

乐观执行：
  共识"很可能成功" → 立即开始执行
  如果最终共识失败 → 回滚
  （快，但需要回滚机制）
```

### 4.2 为什么乐观执行对交易有利

- 大多数情况下共识会成功（恶意节点是少数）
- 乐观执行让用户更快看到结果
- 极少数失败情况才回滚

### 4.3 乐观执行的风险

- 需要可靠的回滚机制
- 如果回滚频繁，反而更慢
- 实现复杂度高

**注**：乐观执行是 Hotstuff 家族的常见优化，HyperBFT 是否采用、以何种形式采用，官方未明确说明。这里是基于性能表现的合理推测。

## 5. 性能参数详解

### 5.1 关键指标

| 指标 | 值 | 说明 |
|------|-----|------|
| 出块间隔 | ~0.1-0.2s | 比 Solana 的 0.4s 还快 |
| 最终性 | ~0.2s（单区块） | 一旦出块即最终 |
| 理论 TPS | ~200,000 | 订单操作 |
| 实际 TPS（高峰） | ~20,000-100,000 | 取决于交易复杂度 |
| 验证者数量 | 16 | 计划扩展 |
| 通信复杂度 | O(n) | Hotstuff 继承 |

### 5.2 为什么能这么快（三个因素）

```
1. 验证者少（16）
   → 网络通信轮次少
   → 投票收集快

2. Hotstuff O(n) 通信 + 门限签名
   → 不会随验证者增加而 n² 爆炸

3. 共识/执行解耦 + 流水线
   → 共识只排序，轻量
   → 多区块并发
```

### 5.3 延迟拆解（推测）

```
用户提交订单 → 验证者收到（~50ms 网络）
验证者验证 → 进入共识提议（~10ms）
共识投票轮次（~100-150ms）
执行/撮合（~1-5ms）
─────────────────────────────
总计：~150-250ms（端到端）
```

## 6. Leader 机制

### 6.1 Leader 的角色

Hotstuff 家族都有 Leader（提议者）：
- 每一轮由一个 Leader 提议区块
- Leader 收集投票，聚合成 QC
- Leader 广播 QC

### 6.2 Leader 轮换

为了防止单个 Leader 作恶或成为瓶颈：
- Leader 按轮次轮换（round-robin 或基于权重）
- 如果 Leader 不响应（超时）→ 触发 view change，换下一个 Leader

### 6.3 Leader 是中心化点吗

**是，但有限**：
- 单轮内 Leader 有提议权（可以选择交易顺序）
- 但 Leader 不能伪造投票（需要 2f+1 个真实投票）
- Leader 轮换限制了单个 Leader 的权力
- Hyperliquid 声称"确定性排序"，意味着 Leader 的排序自由被限制（推测：按接收时间或某种确定性规则排序）

---

## 7. View Change（视图切换）

### 7.1 什么时候触发

当 Leader 失效时（崩溃、作恶、网络分区）：
- 其他节点超时未收到有效提议
- 触发 view change → 换 Leader

### 7.2 Hotstuff View Change 的优势

PBFT 的 view change 是 O(n²) 的复杂操作（这是 PBFT 的痛点之一）。

Hotstuff 把 view change 简化为"正常流程的一部分"：
- 不需要特殊的 view change 协议
- 新 Leader 直接基于最新的 QC 继续
- 复杂度仍是 O(n)

这是 Hotstuff 相对 PBFT/Tendermint 的一个关键优势。

## 8. HyperBFT vs 其他共识的实测对比

| 维度 | HyperBFT | CometBFT (dYdX) | Solana TowerBFT | 
|------|----------|-----------------|-----------------|
| 共识家族 | Hotstuff | Tendermint | PoH + BFT |
| 通信复杂度 | O(n) | O(n²) | O(n) |
| 出块间隔 | ~0.1-0.2s | ~1-6s | 0.4s |
| 最终性 | 0.2s | 6s | ~12.8s（升级前） |
| 验证者 | 16 | ~60 | ~1500 |
| 为什么快 | 少验证者+流水线 | - | PoH 预排序 |

**关键观察**：
- HyperBFT 和 Solana 都是 O(n)，但 HyperBFT 最终性更快（0.2s vs 12.8s）
- 原因：Solana 的 TowerBFT 需要多轮确认，HyperBFT 单区块最终
- 代价：HyperBFT 只有 16 个验证者，Solana 有 1500 个

## 9. 设计权衡总结

### 9.1 HyperBFT 得到了什么

- ✅ 亚秒最终性（0.2s）
- ✅ 高吞吐（流水线 + 解耦）
- ✅ 确定性执行（为无 MEV 铺路）
- ✅ O(n) 通信（理论上可扩展验证者）

### 9.2 HyperBFT 付出了什么

- ❌ 验证者数量少（16）→ 中心化
- ❌ Leader 机制 → 单点提议权
- ❌ 代码不开源 → 无法独立验证
- ❌ 为扩展验证者会牺牲延迟

### 9.3 核心判断

> HyperBFT 的设计是"把 Hotstuff 推到延迟的极致"。它证明了链上共识可以做到亚秒最终性。但代价是验证者数量必须保持很少，这是它和"去中心化"理想之间的根本张力。
