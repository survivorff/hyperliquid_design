# Hyperliquid 交易机制：永续合约、订单类型、资金费率与清算

> 一个 CEX 级的链上永续合约交易所，到底怎么交易？这篇从用户视角出发，拆解 Hyperliquid 的完整交易机制。

---

## 1. 永续合约基础

### 1.1 什么是永续合约

永续合约（Perpetual Futures）是没有到期日的期货合约。传统期货有交割日（比如"BTC 6月合约"），到期必须平仓或交割。永续合约去掉了到期日，用**资金费率（Funding Rate）**机制让合约价格锚定现货。

**为什么加密圈偏爱永续**：
- 不用换月（传统期货每月/每季到期，要移仓）
- 可以用高杠杆（5x-100x）
- 多空双向（看涨做多，看跌做空）
- 资本效率高（$1000 保证金可以控制 $40,000 的仓位）

**Hyperliquid 的永续合约**：
- 150+ 交易对（BTC、ETH、SOL、各种 altcoin、meme coin）
- 最高 40x 杠杆（BTC/ETH），其他币种 20x 或更低
- USDC 计价和结算（所有盈亏用 USDC 计算）
- 全链上 CLOB 撮合（不是 AMM，不是链下撮合）


### 1.2 永续 vs 现货 vs 传统期货

| 维度 | 现货 | 传统期货 | 永续合约 |
|------|------|---------|---------|
| 到期日 | 无 | 有（月/季） | **无** |
| 杠杆 | 1x（或借贷杠杆） | 5-20x | **5-100x** |
| 方向 | 只能做多 | 多空 | **多空** |
| 结算 | 即时 | 到期结算 | **实时盈亏** |
| 价格锚定 | N/A | 到期收敛 | **资金费率** |
| 资本效率 | 低 | 高 | **最高** |

### 1.3 Hyperliquid 的永续合约特点

**和 CEX 永续的对比**：

| 特性 | Binance Futures | Hyperliquid |
|------|----------------|-------------|
| 撮合方式 | 中心化 CLOB | **链上 CLOB** |
| 最大杠杆 | 125x | 40x（BTC/ETH） |
| 结算货币 | USDT/USDC | **USDC** |
| 资金费率 | 每 8 小时 | **每 8 小时**（实时累积） |
| 订单类型 | 全部 | **全部**（限价、市价、止损等） |
| 自托管 | 否 | **是** |
| MEV 风险 | 无 | **无**（确定性排序） |
| 交易确认 | 即时 | **~200ms** |

**关键差异**：Hyperliquid 的杠杆上限比 CEX 低（40x vs 125x），这是**有意的风险控制**。更高杠杆意味着更容易被清算，对系统的 Insurance Fund 压力更大。在链上环境中，清算的可靠性比 CEX 更难保证，所以保守的杠杆上限是合理的。

## 2. 订单类型

Hyperliquid 支持和 CEX 几乎完全一样的订单类型。这是它"CEX 级体验"的核心体现。

### 2.1 基础订单类型

| 订单类型 | 说明 | 适用场景 |
|---------|------|---------|
| **Market（市价单）** | 以当前最优价格立即成交 | 快速进出场 |
| **Limit（限价单）** | 指定价格，等待成交 | 精确入场、做市 |
| **Stop Market（止损市价）** | 触发价到达后以市价成交 | 止损 |
| **Stop Limit（止损限价）** | 触发价到达后挂限价单 | 精确止损 |
| **Take Profit（止盈）** | 盈利到达目标后自动平仓 | 锁定利润 |
| **Trailing Stop（跟踪止损）** | 止损价随价格移动 | 趋势跟踪 |

### 2.2 高级订单参数

**Time-in-Force（有效期）**：

| 参数 | 全称 | 说明 |
|------|------|------|
| **GTC** | Good Till Cancel | 挂单直到成交或手动取消（默认） |
| **IOC** | Immediate Or Cancel | 立即成交能成交的部分，剩余取消 |
| **FOK** | Fill Or Kill | 必须全部成交，否则整单取消 |
| **ALO** | Add Liquidity Only | 只做 Maker（如果会立即成交则取消） |

**Post-Only（只做 Maker）**：
- 确保订单只会作为 Maker 挂在订单簿上
- 如果提交时会立即和对手方成交 → 订单被拒绝
- 做市商必备：保证享受 Maker 费率（0.01%）而非 Taker 费率（0.035%）

**Reduce-Only（只减仓）**：
- 订单只能减少现有持仓，不能增加
- 防止"止损单变成反向开仓"的意外情况
- 止损和止盈订单通常设为 Reduce-Only

### 2.3 TWAP（时间加权平均价格）

Hyperliquid 支持 TWAP 订单：
- 把大单拆成多个小单，在指定时间内均匀执行
- 减少大单对市场的冲击
- 适合机构和大户的大额建仓/平仓

**TWAP 参数**：
- 总数量
- 执行时间窗口（比如 30 分钟）
- 每次执行的间隔
- 价格限制（可选）

### 2.4 和 CEX 订单类型的对比

| 功能 | Binance | dYdX V4 | GMX V2 | Hyperliquid |
|------|---------|---------|--------|-------------|
| 限价单 | ✅ | ✅ | ❌ | ✅ |
| 止损/止盈 | ✅ | ✅ | ✅ | ✅ |
| 跟踪止损 | ✅ | ❌ | ❌ | ✅ |
| TWAP | ✅ | ❌ | ❌ | ✅ |
| Post-Only | ✅ | ✅ | N/A | ✅ |
| IOC/FOK | ✅ | ✅ | N/A | ✅ |

GMX 不支持限价单是因为它用 Oracle 定价，不是订单簿。dYdX V4 功能较全但缺少 TWAP 和跟踪止损。**Hyperliquid 是链上永续里订单类型最完整的**。

## 3. 杠杆与保证金

### 3.1 杠杆倍数

Hyperliquid 的杠杆上限因币种而异：

| 币种 | 最大杠杆 | 初始保证金率 | 维持保证金率 |
|------|---------|------------|------------|
| BTC | 40x | 2.5% | 1.25% |
| ETH | 40x | 2.5% | 1.25% |
| SOL | 20x | 5% | 2.5% |
| 主流 Altcoin | 20x | 5% | 2.5% |
| 小币种 | 5-10x | 10-20% | 5-10% |
| Meme 币 | 3-5x | 20-33% | 10-16% |

**为什么不同币种杠杆不同**：
- 波动率越高 → 杠杆越低
- 流动性越差 → 杠杆越低
- BTC/ETH 波动相对小、流动性深 → 可以给 40x
- Meme 币一天涨跌 50% 很正常 → 只给 3-5x

**杠杆和保证金的关系**：

```
初始保证金 = 持仓名义价值 / 杠杆倍数

例：10x 杠杆做多 1 BTC（价格 $60,000）
初始保证金 = $60,000 / 10 = $6,000
```

### 3.2 保证金模式

**交叉保证金（Cross Margin）**：

- 所有持仓共享一个保证金池
- 账户里所有可用余额都作为保证金
- 一个持仓的未实现盈利可以支撑另一个持仓的亏损
- **优势**：资金效率最高，不容易被单个仓位清算
- **风险**：一个仓位爆仓可能连带所有仓位

```
Cross Margin 示例：
账户余额：$10,000
持仓 A：BTC 多 10x，占用 $3,000 保证金
持仓 B：ETH 空 5x，占用 $2,000 保证金
可用保证金：$10,000 - $3,000 - $2,000 = $5,000
→ 如果 BTC 下跌，$5,000 可用余额 + ETH 空头盈利 都在保护 BTC 多头
```

**逐仓保证金（Isolated Margin）**：

- 每个持仓有独立的保证金
- 一个仓位最多亏完它自己的保证金
- 其他仓位和账户余额不受影响
- **优势**：风险隔离，单仓爆仓不影响全局
- **风险**：资金效率低，更容易被单个仓位清算

```
Isolated Margin 示例：
账户余额：$10,000
持仓 A：BTC 多 10x，分配 $3,000 保证金（isolated）
持仓 B：ETH 空 5x，分配 $2,000 保证金（isolated）
→ 如果 BTC 下跌，最多亏 $3,000，ETH 仓位和剩余 $5,000 不受影响
```

### 3.3 保证金率计算

**初始保证金率（Initial Margin）**：开仓时需要的最低保证金比例。

```
初始保证金率 = 1 / 杠杆倍数
10x 杠杆 → 初始保证金率 = 10%
40x 杠杆 → 初始保证金率 = 2.5%
```

**维持保证金率（Maintenance Margin）**：持仓期间需要维持的最低保证金比例。通常是初始保证金率的一半。

```
维持保证金率 ≈ 初始保证金率 / 2
10x 杠杆 → 维持保证金率 ≈ 5%
40x 杠杆 → 维持保证金率 ≈ 1.25%
```

**当保证金率低于维持保证金率时 → 触发清算**。

### 3.4 未实现盈亏（Unrealized PnL）

```
多头 PnL = (当前标记价格 - 开仓均价) × 持仓数量
空头 PnL = (开仓均价 - 当前标记价格) × 持仓数量

例：10x 做多 1 BTC @ $60,000
BTC 涨到 $61,000 → PnL = +$1,000（相对 $6,000 保证金 = +16.7%）
BTC 跌到 $59,000 → PnL = -$1,000（相对 $6,000 保证金 = -16.7%）
```

**杠杆放大效应**：10x 杠杆意味着价格变动 1% → 盈亏变动 10%。

## 4. 资金费率（Funding Rate）

### 4.1 为什么需要资金费率

永续合约没有到期日，所以没有"到期时价格收敛到现货"的自然机制。资金费率是人为设计的锚定机制：

- 永续价格 > 现货价格（多头过多）→ 多头付钱给空头 → 抑制做多
- 永续价格 < 现货价格（空头过多）→ 空头付钱给多头 → 抑制做空

**效果**：资金费率让永续价格始终围绕现货价格波动，不会长期偏离。

### 4.2 Hyperliquid 的资金费率机制

**结算周期**：每 8 小时结算一次（和 Binance 一样）
- 结算时间：UTC 00:00、08:00、16:00

**计算公式**：

```
Funding Rate = Average Premium Index + clamp(Interest Rate - Average Premium Index, -0.05%, 0.05%)

其中：
- Premium Index = (永续中间价 - Oracle 价格) / Oracle 价格
- Average Premium Index = 过去 8 小时 Premium Index 的时间加权平均
- Interest Rate = 0.01%（固定，代表 USDC 和 crypto 的利率差）
- clamp 函数限制在 [-0.05%, 0.05%] 范围内
```

**简化理解**：
- 资金费率 ≈ 永续价格相对现货的偏离程度
- 偏离越大 → 费率越高 → 纠正力度越大

### 4.3 资金费率的支付

```
资金费用 = 持仓名义价值 × 资金费率

例：持有 10 BTC 多头（名义价值 $600,000），资金费率 = 0.01%
每 8 小时支付：$600,000 × 0.01% = $60
每天支付：$60 × 3 = $180
年化成本：$180 × 365 = $65,700（约 11% 年化）
```

**关键点**：
- 资金费率是**持仓成本**，不是交易手续费
- 长期持仓时，资金费率可能比手续费贵得多
- 牛市时资金费率通常为正（多头付空头）→ 做空有"收租"效应
- 熊市时资金费率通常为负（空头付多头）→ 做多有"收租"效应

### 4.4 资金费率的实时累积

虽然每 8 小时结算一次，但 Hyperliquid 的资金费率是**实时累积**的：

- 你不需要在结算时刻持有仓位才付/收资金费
- 持仓期间每一秒都在累积
- 结算时刻只是"结算"已累积的金额

**这意味着**：你不能通过"结算前平仓、结算后开仓"来逃避资金费率。

### 4.5 资金费率和交易策略

**Funding Rate Arbitrage（资金费率套利）**：

```
策略：当资金费率很高时（比如 0.1%/8h）
1. 在 Hyperliquid 做空（收取资金费率）
2. 在现货市场做多（对冲方向风险）
3. 净收益 = 资金费率收入 - 现货持仓成本

年化收益可达 30-100%（极端行情时）
但风险是：资金费率可能突然反转
```

**对比各平台资金费率**：

| 平台 | 结算周期 | 费率上限 | 特点 |
|------|---------|---------|------|
| Binance | 8h | ±2% | 最大流动性 |
| Hyperliquid | 8h | 无硬上限 | 链上透明 |
| dYdX V4 | 1h | ±0.75% | 更频繁结算 |
| GMX V2 | 实时 | 动态 | 基于 OI 不平衡 |

## 5. 清算机制

### 5.1 什么时候触发清算

当账户的**保证金率**低于**维持保证金率**时，触发清算（Liquidation）。

```
保证金率 = (账户权益) / (持仓名义价值)
账户权益 = 账户余额 + 未实现盈亏

当 保证金率 < 维持保证金率 → 清算

例：
开仓：10x 做多 1 BTC @ $60,000，保证金 $6,000
维持保证金率：5%（即需要维持 $3,000）
清算价格 ≈ $60,000 × (1 - 1/10 + 维持保证金率)
         ≈ $60,000 × (1 - 0.10 + 0.05)
         ≈ $57,000
→ 当 BTC 跌到 ~$57,000 时触发清算
```

### 5.2 标记价格（Mark Price）

清算基于**标记价格**而非最新成交价。这是为了防止"插针清算"（通过瞬间操纵价格触发大量清算）。

**标记价格的计算**：

```
Mark Price = Oracle Price × (1 + Funding Basis)

其中：
- Oracle Price = 多个外部交易所的加权中间价
- Funding Basis = 当前累积的资金费率偏差
```

**为什么用标记价格**：
- 最新成交价可能被操纵（一笔大单就能瞬间拉/砸价格）
- Oracle 价格反映全市场共识，更难操纵
- 防止"自我清算攻击"（在自己的平台上操纵价格清算别人）

### 5.3 清算流程

```
1. 监控：清算引擎持续监控所有持仓的保证金率
2. 触发：标记价格导致某账户保证金率 < 维持保证金率
3. 接管：清算引擎接管该账户的持仓
4. 平仓：在市场上以限价单方式平仓
   - 价格 = 破产价格（Bankruptcy Price）附近
   - 如果市场有足够流动性 → 正常平仓
5. 剩余处理：
   - 如果平仓价格好于破产价 → 差额进入 Insurance Fund
   - 如果平仓价格差于破产价 → Insurance Fund 补贴差额
6. 最后手段：如果 Insurance Fund 不足 → 触发 ADL
```

### 5.4 Insurance Fund（保险基金）

Insurance Fund 是清算机制的安全垫：

**资金来源**：
- 清算罚金（清算时收取的额外费用）
- 清算盈余（平仓价格好于破产价的差额）
- 平台收入的一部分

**资金用途**：
- 覆盖清算亏损（平仓价格差于破产价时）
- 防止 ADL 触发

**Hyperliquid 的 Insurance Fund 特点**：
- 链上透明可查
- 由 HLP Vault 部分承担（HLP 是 Insurance Fund 的"后备军"）
- 规模随平台增长而增长

### 5.5 ADL（自动减仓）

ADL 是**最后手段**，当 Insurance Fund 不足以覆盖清算亏损时触发。

**机制**：
- 系统找到**盈利最多且杠杆最高**的对手方
- 自动减少他们的持仓（强制部分平仓）
- 被 ADL 的用户收到通知，按当时的标记价格结算

**ADL 优先级排序**：

```
ADL 优先级 = 未实现盈利率 × 有效杠杆

盈利越多 + 杠杆越高 → 越先被 ADL
```

**为什么这么设计**：
- 盈利多的人"赚够了"，被减仓的痛感相对小
- 高杠杆的人对系统风险贡献大，优先被减
- 这比"全部用户按比例分摊"更公平

**ADL vs 社会化亏损**：

| 方式 | 说明 | 用谁 |
|------|------|------|
| **ADL** | 只减仓盈利最多的对手方 | Hyperliquid、Binance |
| **社会化亏损** | 所有盈利用户按比例分摊 | 早期 BitMEX |
| **Insurance Fund 兜底** | 基金覆盖所有亏损 | 理想情况 |

Hyperliquid 的优先级：Insurance Fund → HLP Vault → ADL。正常情况下 ADL 极少触发。

### 5.6 JELLY 事件：清算机制的极端测试

2025 年 3 月的 JELLY 事件是 Hyperliquid 清算机制的一次极端压力测试（详见 08-risks-and-events.md）。

**简要回顾**：
- 攻击者在低流动性的 JELLY 永续市场建立巨额空头
- 在现货市场拉升 JELLY 价格
- 主动触发自己的清算
- 清算引擎无法在市场上平仓（没有对手方）
- HLP Vault 被迫接手，浮亏 ~$1000 万
- 最终验证者投票下架该市场

**教训**：
- 清算机制在**流动性充足**的市场运行良好
- 在**流动性极差**的市场，清算可能变成系统性风险
- HLP 作为"最后接盘方"的角色需要更严格的风险限制
- 事后 Hyperliquid 加强了上架标准和 HLP 风险参数

## 6. 费用结构

### 6.1 交易手续费

| 层级 | Maker 费率 | Taker 费率 | 条件 |
|------|-----------|-----------|------|
| 标准 | 0.010% | 0.035% | 默认 |
| VIP 1 | 0.008% | 0.030% | 14天交易量 > $5M |
| VIP 2 | 0.006% | 0.025% | 14天交易量 > $25M |
| VIP 3 | 0.004% | 0.020% | 14天交易量 > $100M |
| VIP 4 | 0.002% | 0.017% | 14天交易量 > $500M |
| MM | 0.000% | 0.015% | 做市商计划（需申请） |

**Maker vs Taker**：
- **Maker**：挂限价单提供流动性（订单在簿上等待成交）
- **Taker**：市价单或立即成交的限价单（消耗流动性）
- Maker 费率远低于 Taker → 激励提供流动性

### 6.2 费用对比

| 平台 | Maker | Taker | 说明 |
|------|-------|-------|------|
| **Hyperliquid** | 0.010% | 0.035% | 链上最低 |
| dYdX V4 | 0.020% | 0.050% | |
| GMX V2 | 0.050% | 0.070% | 含价格影响 |
| Binance Futures | 0.020% | 0.050% | VIP 0 |
| Bybit | 0.020% | 0.055% | |

**Hyperliquid 的费率优势**：
- Maker 费率是 dYdX 的一半
- Taker 费率是 dYdX 的 70%
- 对高频做市商来说，这个差距意味着**每天数万美元的成本差异**

### 6.3 其他费用

| 费用类型 | 金额 | 说明 |
|---------|------|------|
| 开仓/平仓 | 包含在手续费中 | 无额外费用 |
| 资金费率 | 动态 | 每 8 小时结算（见第 4 节） |
| 清算罚金 | ~0.5-1% | 被清算时额外扣除 |
| 提现 | 免费 | 提现到 Arbitrum 不收费 |
| Gas 费 | 免费 | 下单/撤单不消耗 Gas |

**Gas-free 的意义**：
- 做市商每秒下/撤几千个订单 → 如果每次都付 Gas，成本不可承受
- 这是 Hyperliquid 流动性深的根本原因之一
- 对比 Solana 上的 DEX：每次下单/撤单都要付 ~$0.001-0.01 Gas

### 6.4 Builder Codes（推荐码）

Builder Codes 是 Hyperliquid 的推荐/分佣系统：
- 开发者/KOL 创建 Builder Code
- 用户通过 Builder Code 交易
- 开发者获得该用户手续费的一部分（通常 10-40%）
- 用户可能获得手续费折扣

**这是 Hyperliquid 生态增长的重要机制**：前端聚合器、交易机器人、社区 KOL 都通过 Builder Codes 获得收入。

## 7. 现货交易

### 7.1 现货 vs 永续

Hyperliquid 在 2024 年底上线了现货交易，和永续共享同一个订单簿引擎。

| 维度 | 永续合约 | 现货交易 |
|------|---------|---------|
| 杠杆 | 3-40x | 1x（无杠杆） |
| 方向 | 多空 | 只能买卖 |
| 资金费率 | 有 | 无 |
| 结算 | USDC | Token 直接交割 |
| 清算 | 有 | 无 |
| 保证金 | 需要 | 全额支付 |

### 7.2 HIP-1 Token 标准

Hyperliquid 现货交易的 Token 使用 **HIP-1** 标准（不是 ERC-20）：

- HIP-1 是 HyperCore 的原生 Token 标准
- 铸造、转账、销毁都是原生操作（不烧 EVM Gas）
- 比 ERC-20 更快、更便宜
- 可以通过桥和 HyperEVM 上的 ERC-20 互转

### 7.3 现货上架机制

新 Token 上架 Hyperliquid 现货需要通过**拍卖**：

- 每 31 小时一次拍卖
- 项目方出价竞拍上架名额
- 最高出价者获得上架权
- 拍卖收入归平台（进入 Assistance Fund）

**拍卖价格**：从几万到几十万美元不等，取决于市场热度。

### 7.4 现货作为永续保证金

Hyperliquid 的一个独特功能：**现货持仓可以作为永续合约的保证金**。

```
例：
你持有 10 ETH 现货（价值 $30,000）
你可以用这 10 ETH 作为保证金，开 BTC 永续合约
→ 不需要额外存入 USDC
→ 资本效率大幅提升
```

这类似 CEX 的"统一账户"（Portfolio Margin），但在链上实现。

## 8. Hyperps（预发行永续）

### 8.1 什么是 Hyperps

Hyperps 是 Hyperliquid 的"预发行永续合约"：对**尚未正式上线**的 Token 开设永续合约。

**典型场景**：
- 某个项目宣布要发 Token，但还没 TGE（Token Generation Event）
- Hyperliquid 开设该 Token 的 Hyperps
- 交易者可以提前做多/做空，表达对 Token 价格的预期

### 8.2 Hyperps 的定价

由于没有现货市场，Hyperps 的定价方式不同：
- 不能用"永续价格 vs 现货价格"来计算资金费率
- 改用 **Oracle 价格**或**预估价格**作为锚定
- 资金费率基于市场 OI（未平仓合约）的多空比例

### 8.3 风险特征

Hyperps 的风险远高于普通永续：

| 风险 | 说明 |
|------|------|
| 流动性极差 | 没有现货市场支撑，盘口很薄 |
| 价格操纵 | 没有深度现货市场锚定，容易被操纵 |
| 杠杆低 | 通常只有 3-5x |
| 结算不确定 | Token 可能最终不上线，或价格和预期差很远 |

**适合谁**：
- 对项目有深入研究的交易者
- 想提前布局的投资者
- 高风险偏好的投机者

## 9. 一笔交易的完整生命周期

### 9.1 从下单到成交

```
用户操作                    系统处理
─────────                  ─────────
1. 选择交易对（BTC-PERP）
2. 选择方向（做多）
3. 选择杠杆（10x）
4. 输入数量（1 BTC）
5. 选择订单类型（限价 $60,000）
6. 点击下单
        │
        ▼
7. 前端构造订单对象
8. EIP-712 签名（用户钱包）
        │
        ▼
9. 发送到 Hyperliquid 验证者
10. 验证签名 + 检查保证金
11. 进入 HyperBFT 共识（~200ms）
12. 订单进入链上订单簿
        │
        ▼
13a. 如果有匹配 → 立即撮合
     → 更新双方余额和持仓
     → WebSocket 推送成交通知
        
13b. 如果无匹配 → 挂在簿上等待
     → 直到有对手方或用户取消
```

### 9.2 持仓管理

开仓后，用户可以：
- **加仓**：增加同方向持仓
- **减仓**：部分平仓
- **平仓**：完全关闭持仓
- **设置止损/止盈**：条件单
- **调整杠杆**：增加/减少保证金
- **切换保证金模式**：Cross ↔ Isolated

### 9.3 盈亏结算

```
已实现盈亏（平仓时）：
  多头：(平仓价 - 开仓价) × 数量 - 手续费 - 累积资金费率
  空头：(开仓价 - 平仓价) × 数量 - 手续费 - 累积资金费率

例：
  10x 做多 1 BTC @ $60,000
  平仓 @ $62,000
  持仓 3 天，资金费率累计 0.03%
  
  毛利润 = ($62,000 - $60,000) × 1 = $2,000
  手续费 = $60,000 × 0.035% + $62,000 × 0.035% = $42.7
  资金费率 = $60,000 × 0.03% = $18
  净利润 = $2,000 - $42.7 - $18 = $1,939.3
  ROI = $1,939.3 / $6,000（保证金）= 32.3%
```

### 10.1 全维度对比

### 10.2 各平台的 Tradeoff

**Hyperliquid**：
- ✅ 链上最快、费率最低、订单类型最全
- ❌ 杠杆上限较低（40x）、验证者中心化风险

**dYdX V4**：
- ✅ 更去中心化（60 验证者）、更频繁的资金费率结算
- ❌ 性能不如 Hyperliquid、生态较弱

**GMX V2**：
- ✅ 高杠杆（100x）、无需做市商
- ❌ 无价格发现、LP 风险、无限价单

**Binance Futures**：
- ✅ 最高杠杆、最深流动性、即时确认
- ❌ 中心化、不自托管、审查风险

## 11. 交易机制的设计哲学

### 11.1 为什么 Hyperliquid 的交易机制能成功

回顾整个交易机制设计，Hyperliquid 的成功来自几个关键决策：

**1. 选择 CLOB 而非 AMM**
- CLOB 有真正的价格发现
- 做市商可以精确管理风险
- 资本效率远高于 AMM
- 代价：需要自建高性能链

**2. Gas-free 订单**
- 做市商可以高频报价
- 流动性深度接近 CEX
- 代价：需要特殊的状态管理（订单在内存中，只有成交才上链）

**3. 保守的杠杆上限**
- 40x 而非 125x
- 减少系统性清算风险
- 减少 Insurance Fund 压力
- 代价：部分高杠杆交易者流失到 CEX

**4. 完整的订单类型**
- 和 CEX 功能对齐
- 降低用户迁移成本
- 代价：工程复杂度高

### 11.2 未来可能的演进

**1. 更多交易对**
- HIP-3 开放了 permissionless 上架
- 未来可能有数千个交易对
- 包括传统资产（股票、商品、外汇）的永续

**2. 期权产品**
- HIP-4 计划引入固定期限合约
- 为期权产品铺路
- 和永续共享保证金

**3. 更高杠杆（可能）**
- 随着 Insurance Fund 增长和清算机制成熟
- 可能逐步提高杠杆上限
- 但需要平衡风险

**4. 跨链保证金**
- 目前只支持 USDC 和 Hyperliquid 原生资产作为保证金
- 未来可能支持其他链的资产（通过桥接）
