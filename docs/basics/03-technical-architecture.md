# Hyperliquid 技术架构：HyperBFT + HyperCore + HyperEVM

> 一条为交易定制的 L1 链，三层架构各司其职。这篇拆解每一层的设计决策和工程实现。

---

## 1. 架构总览

Hyperliquid 的技术栈是**垂直整合**的三层：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: HyperEVM（通用智能合约层）                      │
│  - 完全兼容 EVM                                          │
│  - 可以和 HyperCore 原子交互                              │
│  - DeFi 协议部署在这里                                    │
└─────────────────────────────────────────────────────────┘
                         ↕ 原子组合
┌─────────────────────────────────────────────────────────┐
│  Layer 2: HyperCore（原生交易引擎）                       │
│  - 全链上 CLOB（订单簿）                                  │
│  - 永续合约 + 现货交易                                    │
│  - 保证金、清算、资金费率                                  │
│  - Gas-free 订单（下单/撤单不消耗 Gas）                    │
└─────────────────────────────────────────────────────────┘
                         ↕ 确定性执行
┌─────────────────────────────────────────────────────────┐
│  Layer 1: HyperBFT（共识层）                              │
│  - Hotstuff 变体，优化为高吞吐低延迟                       │
│  - 16 个验证者（计划扩展）                                 │
│  - 0.2 秒最终性                                           │
│  - 200,000 TPS 理论上限                                   │
└─────────────────────────────────────────────────────────┘
```

**关键设计哲学**：不是"在通用链上做应用"，是"为应用造一条链"。每一层都为交易场景优化。

## 2. Layer 1：HyperBFT 共识

### 2.1 什么是 HyperBFT

HyperBFT 是 Hyperliquid 自研的共识算法，基于 **Hotstuff** 协议的改进版本。

**Hotstuff 是什么**：
- 2019 年由 VMware Research 提出的 BFT 共识协议
- Facebook（Meta）的 Libra/Diem 区块链用的就是 Hotstuff 变体
- 核心优势：**线性通信复杂度**（O(n) 而非传统 BFT 的 O(n²)）

**HyperBFT 的改进**：
- **解耦架构**：共识和执行分离（consensus 和 execution 在不同线程）
- **流水线化**：多个区块可以同时处于不同阶段（propose → vote → commit）
- **乐观执行**：在共识完成前就开始执行交易（如果最终共识不同则回滚）

### 2.2 性能参数

| 参数 | 值 | 对比 |
|------|-----|------|
| 理论 TPS | 200,000 | Solana ~4000 实际，Ethereum ~15 |
| 实际 TPS（高峰） | ~50,000-100,000 | 取决于交易复杂度 |
| 最终性 | **0.2 秒**（单区块确认） | Solana ~0.4s，Cosmos ~6s |
| 验证者数量 | 16 | Solana ~1500，Ethereum ~900K |
| 通信复杂度 | O(n) | 传统 BFT O(n²) |

### 2.3 为什么只有 16 个验证者

这是 Hyperliquid 最受争议的设计决策。

**原因**：
- BFT 共识的延迟和验证者数量正相关
- 16 个验证者 → 0.2 秒最终性
- 100 个验证者 → 可能 1-2 秒最终性
- 对交易所来说，**延迟比去中心化更重要**

**风险**：
- 16 个验证者中 6 个串通就能审查交易（BFT 容忍 < 1/3 恶意）
- 验证者身份不完全公开
- 和"去中心化"的叙事有矛盾

**Hyperliquid 的回应**：
- 计划逐步增加验证者数量
- 引入 HYPE 质押机制（持有者可以委托给验证者）
- 长期目标是"足够去中心化"而非"最大化去中心化"

### 2.4 确定性执行

HyperBFT 的一个关键特性：**交易执行是确定性的**。

这意味着：
- 你提交一笔订单，它要么被执行，要么被拒绝
- **不存在"交易被 revert"的情况**（和 EVM 不同）
- 不存在 MEV 攻击（因为排序是确定的，不能被重排）

**对比 EVM**：
- EVM 上交易可能因为 Gas 不足、状态变化等原因 revert
- EVM 上 MEV 搜索者可以重排交易顺序获利
- Hyperliquid 上这些都不存在

**实现方式**：
- 订单在进入共识前就被验证（余额够不够、保证金够不够）
- 不合法的订单直接被拒绝，不进入区块
- 进入区块的订单**保证执行成功**

## 3. Layer 2：HyperCore（交易引擎）

### 3.1 全链上订单簿（On-chain CLOB）

**这是 Hyperliquid 最核心的技术创新**。

传统理解：订单簿不可能放在链上（太慢、太贵）。
Hyperliquid 的回答：**如果链本身就是为订单簿设计的呢？**

**HyperCore 的订单簿特性**：

| 特性 | 说明 |
|------|------|
| 全链上 | 每个限价单都是链上状态 |
| Gas-free 下单 | 下单和撤单**不消耗 Gas** |
| 确定性撮合 | 撮合结果在提交时就确定 |
| 价格-时间优先 | 和 CEX 一样的撮合规则 |
| 支持所有订单类型 | 限价、市价、止损、止盈、IOC、FOK、GTC |

**为什么下单不消耗 Gas**：

在 EVM 链上，每个状态变化都要付 Gas。但 Hyperliquid 的设计是：
- 下单 = 把订单放入内存中的订单簿（不改变持久状态）
- 撤单 = 从内存中移除（同样不改变持久状态）
- **只有成交时**才改变链上状态（余额变化、持仓变化）

这意味着做市商可以**每秒下/撤几千个订单**而不付任何费用。这是 Hyperliquid 流动性深的根本原因。

### 3.2 撮合引擎

**撮合规则**：价格优先 → 时间优先（和 Nasdaq / NYSE 一样）

**撮合流程**：

```
1. 用户提交订单（签名 + 发送到验证者）
2. 验证者验证订单合法性（余额、保证金）
3. 订单进入 HyperBFT 共识
4. 共识达成后，订单进入订单簿
5. 如果有匹配的对手方 → 立即撮合
6. 撮合结果写入链上状态（余额变化）
7. 如果没有匹配 → 订单挂在簿上等待
```

**延迟拆解**：

```
用户发送订单 → 验证者收到（~50ms 网络延迟）
验证者验证 → 进入共识（~10ms）
共识达成（~100-200ms）
撮合执行（~1ms）
总计：~150-250ms 从提交到确认
```

这个延迟和 Binance Futures（~10-50ms）还有差距，但已经是链上最快的。

### 3.3 保证金系统

Hyperliquid 支持两种保证金模式：

**交叉保证金（Cross Margin）**：
- 所有持仓共享一个保证金池
- 一个持仓的盈利可以支撑另一个持仓的亏损
- 资金效率高，但风险也高（一个爆仓可能连锁）

**逐仓保证金（Isolated Margin）**：
- 每个持仓有独立的保证金
- 一个持仓爆仓不影响其他
- 资金效率低，但风险隔离

### 3.4 清算机制

当用户的保证金率低于维持保证金要求时，触发清算：

```
维持保证金率 = 持仓价值 × 维持保证金比例
当前保证金 < 维持保证金 → 触发清算
```

**清算流程**：

1. **标记价格**（Mark Price）触发清算条件
2. 清算引擎接管持仓
3. 尝试在市场上平仓（限价单）
4. 如果市场流动性不足 → **ADL（自动减仓）**
5. 清算罚金进入 **Insurance Fund**

**ADL（Auto-Deleveraging）**：
- 当 Insurance Fund 不足以覆盖亏损时
- 系统自动减少盈利最多的对手方的持仓
- 这是最后手段，正常情况下不会触发

### 3.5 资金费率（Funding Rate）

永续合约没有到期日，用**资金费率**让永续价格锚定现货价格：

```
每 8 小时结算一次
资金费率 > 0：多头付给空头（永续价格 > 现货）
资金费率 < 0：空头付给多头（永续价格 < 现货）
```

**Hyperliquid 的资金费率计算**：
- 基于永续价格和 Oracle 价格的偏差
- 加上利率成分
- 每 8 小时结算，但实时累积

## 4. Layer 3：HyperEVM（智能合约层）

### 4.1 为什么需要 EVM 层

HyperCore 是为交易优化的，但它**不是通用计算平台**。你不能在 HyperCore 上部署任意智能合约。

HyperEVM 解决这个问题：
- 完全兼容 EVM（Solidity / Vyper 合约可以直接部署）
- 和 HyperCore **共享状态**
- 可以**原子性地**和订单簿交互

### 4.2 原子组合性（Atomic Composability）

这是 HyperEVM 最独特的能力：

**场景**：一个借贷协议想在清算时自动在 Hyperliquid 订单簿上卖出抵押品。

在其他链上：
- 借贷协议在 Aave（Ethereum）
- 交易在 Hyperliquid（另一条链）
- 两步操作，不是原子的，中间可能价格变化

在 Hyperliquid 上：
- 借贷协议部署在 HyperEVM
- 清算时**在同一笔交易里**调用 HyperCore 的订单簿卖出
- 原子性保证：要么全部成功，要么全部回滚

**这打开了什么可能**：
- 结构化产品（期权 + 永续组合）
- 自动化策略（条件触发 + 订单执行）
- 借贷 + 交易一体化（用持仓做抵押借贷）

### 4.3 HyperEVM 的技术实现

- **共享状态**：HyperEVM 和 HyperCore 在同一个验证者网络上运行
- **Precompiles**：HyperEVM 有特殊的预编译合约，可以直接调用 HyperCore 的功能（下单、查询持仓等）
- **Gas 模型**：HyperEVM 的 Gas 用 HYPE Token 支付（和 HyperCore 的 Gas-free 订单不同）

### 4.4 当前 HyperEVM 生态

2025 年 Q2 上线以来，已有多个协议部署：

| 类别 | 代表项目 | 功能 |
|------|---------|------|
| 借贷 | HyperLend, Felix | 用 Hyperliquid 持仓做抵押 |
| 期权 | HyperOptions | 基于永续的期权产品 |
| 收益 | HyperYield | Vault 策略聚合 |
| 稳定币 | USDH | Hyperliquid 原生稳定币 |
| DEX | HyperSwap | HyperEVM 上的现货 AMM |

生态还在早期，但方向清晰：**围绕订单簿构建金融乐高**。

## 5. 数据流：一笔交易的完整路径

```
用户（浏览器/API）
    │
    │ 1. 构造订单 + EIP-712 签名
    ↓
验证者节点
    │
    │ 2. 验证签名 + 检查余额/保证金
    ↓
HyperBFT 共识
    │
    │ 3. 多数验证者投票确认（~200ms）
    ↓
HyperCore 执行
    │
    │ 4. 订单进入订单簿
    │ 5. 如果有匹配 → 撮合 → 更新余额/持仓
    ↓
状态确认
    │
    │ 6. 最终性达成（0.2s）
    │ 7. WebSocket 推送给用户
    ↓
用户看到结果
```

**总延迟**：~200-300ms（从提交到确认）

## 7. 架构的 Tradeoff

### 优势

- ✅ CEX 级性能（延迟、吞吐）
- ✅ 全链上可验证（不是黑盒）
- ✅ 确定性执行（无 MEV、无 revert）
- ✅ Gas-free 订单（做市商友好）
- ✅ 原子组合性（HyperEVM）

### 代价

- ❌ 16 个验证者 → 中心化风险
- ❌ 代码不开源 → 无法独立审计
- ❌ 桥接风险 → 资金从 Arbitrum 桥过来
- ❌ 单点依赖 → 如果 Hyperliquid 链宕机，所有资金被锁
- ❌ 新链生态薄 → 和以太坊的 DeFi 乐高不互通

### 核心判断

Hyperliquid 的架构选择本质上是：

> **"为了交易体验，我愿意牺牲一定程度的去中心化和安全性。"**

这个 tradeoff 是否合理，取决于你对"链上交易所"的期望是什么：
- 如果你要的是"不信任任何人" → Hyperliquid 不够去中心化
- 如果你要的是"比 CEX 更透明但体验一样好" → Hyperliquid 是目前最优解

## 8. 未来架构演进方向

**1. 增加验证者数量**
- 从 16 → 50 → 100+
- 代价：延迟可能从 0.2s 增加到 0.5-1s
- 需要在性能和去中心化之间找新平衡

**2. 代码开源**
- 目前 Hyperliquid 的核心代码不开源
- 社区一直在呼吁开源
- 开源后可以独立审计，增加信任

**3. 多排序器**
- 目前排序器（决定交易顺序的节点）有中心化成分
- 未来可能引入去中心化排序器网络
- 类似 Espresso / Astria 的 shared sequencer 方案

**4. 跨链互操作**
- 目前只能通过 Arbitrum 桥入金
- 未来可能支持更多链的直接桥接
- 或者通过 IBC（如果接入 Cosmos 生态）
