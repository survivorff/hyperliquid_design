# 风险与事件：JELLY 事件、Vault 风险、中心化争议与监管

> 没有完美的系统。Hyperliquid 的成功背后有真实的风险和已经发生的危机。这篇不回避问题，系统梳理 Hyperliquid 面临的所有风险维度。

---

## 1. 风险总览

Hyperliquid 的风险可以分为五个维度：

| 维度 | 严重程度 | 已发生过？ | 核心问题 |
|------|---------|-----------|---------|
| **安全事件** | 高 | 是（JELLY） | 经济攻击、合约漏洞 |
| **中心化风险** | 中-高 | 持续存在 | 16 验证者、代码不开源 |
| **Vault/HLP 风险** | 中 | 是（JELLY） | 中央对手方的尾部风险 |
| **桥接风险** | 高 | 尚未 | 资金桥是单点故障 |
| **监管风险** | 中-高 | 逐步升温 | Permissionless perp 的合规性 |

## 2. JELLY 事件：最严重的一次危机

### 2.1 背景

**JELLYJELLY** 是 2025 年 3 月在 Solana 上线的一个小型 meme 币。Hyperliquid 快速上架了它的永续合约，借 meme 交易热度。

**问题的根源**：
- JELLY 现货流动性极浅（市值仅 ~$50M）
- Hyperliquid 的永续合约 OI 可以远超现货市值
- HLP Vault 是该市场的主要做市方（因为没有专业做市商愿意做）
- 这创造了一个**可被利用的不对称**

### 2.2 攻击时间线

| 时间（UTC） | 事件 |
|------------|------|
| T-24h | 攻击者分散建仓（三个账户），累计 ~$12M 名义价值 JELLY 空头 |
| T-6h | 攻击者在 Solana 上开始买入 JELLY 现货（拉升价格） |
| T-3h | JELLY 现货价格上涨 ~300%，永续价格跟随 |
| T-1h | 攻击者空头浮亏增大，但保证金仍充足 |
| **T+0** | 攻击者**主动提取保证金**，账户保证金率瞬间跌破维持线 |
| T+1min | 清算引擎触发，需要买入大量 JELLY perp 平空 |
| T+5min | 市场上没有对手方 → **HLP Vault 被迫接手** |
| T+15min | JELLY 价格继续被拉升，HLP 浮亏持续扩大 |
| T+30min | HLP 浮亏 ~**$10M**，逼近总资本的 10-15% |
| T+45min | 社区和团队紧急讨论 |
| T+1h | **验证者紧急投票**（99%+ 通过）：下架 JELLY 永续，按危机前价格结算 |
| T+2h | 下架执行完成，所有头寸按预危机价平仓 |

### 2.3 攻击的经济逻辑

```
攻击者的策略：
1. 在 Hyperliquid 开巨额空头（$12M 名义）
2. 在 Solana 现货市场拉升 JELLY（成本很低，因为流动性浅）
3. 主动触发自己的清算
4. 清算引擎无法平仓 → HLP 被迫接手
5. 继续拉升价格 → HLP 亏损
6. 如果 HLP 被迫在高价平仓 → 攻击者在现货端获利

本质：利用"低流动性市场 + HLP 兜底机制"的组合漏洞
```

### 2.4 验证者干预的争议

**支持方观点**：
- 这是紧急情况，不干预 HLP 可能资不抵债
- 投票是公开的（99%+ 通过），不是单方面决定
- 用户资金没有损失（按公平价格结算）
- 类似传统交易所的"熔断机制"

**反对方观点**：
- 验证者可以"回滚交易" → 这和 CEX 有什么区别？
- 16 个验证者投票 = 16 个人决定所有用户的命运
- "去中心化"的承诺在危机时被打破
- 如果下次是 BTC 市场出问题，也会这么干吗？

**我的判断**：
- 干预本身是**正确的**（保护了系统和用户）
- 但干预的**能力**暴露了中心化风险
- 关键不是"该不该干预"，而是"谁有权干预、在什么条件下"
- 需要更明确的"紧急治理框架"

### 2.5 事后改进

| 改进 | 说明 |
|------|------|
| **上架标准收紧** | 新市场需要更高流动性门槛 |
| **HLP 风险参数重构** | 单市场最大敞口限制 |
| **异类市场独立保证金池** | Meme 币和主流市场分开 |
| **断路器机制** | 价格偏离过大时自动暂停 |
| **OI 上限** | 永续 OI 不能超过现货市值的一定比例 |

### 2.6 JELLY 事件的深层教训

1. **HLP 作为中央对手方是单点故障**：任何新的低流动性市场都是新的攻击面
2. **Permissionless 上架 + 兜底机制 = 矛盾**：HIP-3 让任何人上市场，但 HLP 要为所有市场兜底
3. **链上治理的速度和中心化是 tradeoff**：快速响应需要少数人决策
4. **经济攻击比技术攻击更难防**：共识没有 bug，bug 在经济模型

## 3. 中心化风险

### 3.1 验证者集中

| 指标 | Hyperliquid | 对比 |
|------|-------------|------|
| 验证者数量 | **16** | 以太坊 >100万，Solana ~1500 |
| BFT 安全阈值 | 需要 11/16 诚实 | 6 个串通就能审查 |
| 验证者身份 | 部分公开 | 不完全透明 |
| 地理分布 | 集中在北美/欧洲 | 亚洲节点少 |

**具体风险**：
- **审查风险**：6 个验证者串通可以拒绝处理某些交易
- **停机风险**：6 个验证者同时离线可以停止出块
- **监管风险**：如果验证者在美国，可能被要求审查特定地址
- **串谋风险**：验证者之间可能有利益关联

### 3.2 代码不开源

Hyperliquid 的核心代码（HyperBFT、HyperCore）**不开源**。

**这意味着**：
- 社区无法独立审计代码
- 无法验证"确定性执行"的声明
- 无法确认是否有后门
- 安全性完全依赖团队的声誉

**对比**：
- dYdX V4：完全开源
- GMX：完全开源
- Uniswap：完全开源
- Hyperliquid：**不开源**

### 3.3 排序器中心化

虽然有 16 个验证者参与共识，但**交易排序**可能有中心化成分：
- 谁决定交易进入区块的顺序？
- 是否存在"优先处理某些交易"的可能？
- 由于代码不开源，无法独立验证

### 3.4 团队集中

- 核心团队 ~10 人
- 创始人 Jeff Yan 有极大的决策权
- 没有公开的治理框架
- 关键决策由团队主导

## 4. Vault / HLP 风险

### 4.1 HLP 的多重角色

HLP 同时是**做市商 + 清算接盘方 + 保险基金后备**。这三个角色的组合创造了独特的风险。

### 4.2 HLP 的风险来源

| 风险 | 说明 | 严重程度 |
|------|------|---------|
| **单边敞口** | 极端行情下累积大量单方向持仓 | 高 |
| **低流动性市场** | 被迫在没有对手方的市场接盘 | 高 |
| **Oracle 操纵** | 在错误价格接盘 | 中 |
| **系统性风险** | 多个市场同时出问题 | 中-高 |

### 4.3 HLP 存款人须知

- **不是无风险收益**：历史年化 10-30%，但某些月份为负
- **JELLY 事件时浮亏 ~$10M**
- **没有存款保险**
- 建议：不要 all-in，理解尾部风险

## 5. 桥接风险

### 5.1 资金路径

```
用户资金：以太坊 → Arbitrum → Hyperliquid
两次桥接 = 两次风险暴露
```

### 5.2 历史桥接攻击

| 事件 | 损失 | 原因 |
|------|------|------|
| Ronin Bridge | $625M | 验证者私钥被盗 |
| Wormhole | $320M | 合约漏洞 |
| Nomad | $190M | 验证逻辑错误 |

Hyperliquid 的桥还没被攻击过，但桥是加密行业被攻击最多的基础设施。

### 5.3 缓解措施

- 多重签名
- 延迟提现（大额有时间锁）
- 异常交易监控
- 计划支持更多入金通道

## 6. 监管风险

### 6.1 核心问题

| 问题 | 司法管辖区 | 严重程度 |
|------|-----------|---------|
| 永续合约是否是期货 | 美国 CFTC | 高 |
| Token 化股票是否是证券 | 美国 SEC、欧盟 ESMA | 高 |
| KYC/AML 合规 | 全球 | 中-高 |
| Permissionless 上架 | 全球 | 中 |

### 6.2 HIP-3 的监管敏感性

HIP-3 允许上架 Token 化股票、选举永续等 → 可能触发证券法、赌博法。

### 6.3 可能的监管场景

- **温和**：要求 KYC，限制某些市场类型
- **严厉**：禁止向特定国家提供服务，下架敏感市场
- **极端**：定性为非法交易所（可能性低）

### 6.4 Hyperliquid 的准备

- 招聘合规人员
- 探索 zkKYC
- 可能推出"合规版"面向机构

## 7. 技术风险

### 7.1 共识层

- 共识分叉（低概率）
- 活性失败 / 链停止出块（低-中概率）
- 高负载下性能退化（中概率）

### 7.2 HyperEVM

- Precompile 实现漏洞
- 跨层 reentrancy 攻击
- 状态膨胀（长期）

### 7.3 Oracle

- 标记价格被操纵 → 错误清算
- 资金费率异常
- HIP-3 市场的 builder Oracle 不可靠

---

## 8. 经济风险

### 8.1 流动性危机

大量用户同时提现 → 桥拥堵 → HLP 紧急平仓 → 市场流动性枯竭

### 8.2 Token 价格螺旋

HYPE 下跌 → 质押价值降低 → 安全性降低 → 信心下降 → 交易量下降 → 收入下降 → 回购减少 → 价格进一步下降

### 8.3 做市商撤退

主要做市商撤出 → 盘口深度下降 → 滑点增大 → 用户体验恶化 → 交易量下降

## 11. 总结

### 核心判断

> Hyperliquid 的风险是**真实的但可管理的**。它不是完美的去中心化系统，但比 CEX 更透明、比其他 DEX 更实用。

**用户需要理解**：
- "去中心化"不等于"无风险"
- 自托管消除了平台跑路风险，但引入了技术风险
- 高收益（HLP、杠杆交易）伴随高风险
- 监管是最大的外部不确定性

**Hyperliquid 需要做的**：
- 增加验证者（最紧迫）
- 开源代码（建立信任）
- 完善治理框架（明确"谁能在什么条件下干预"）
- 持续改进风控（HLP 分层、断路器、OI 限制）
