# 前世今生：链上永续合约的进化史

> 从 2020 年到 2026 年，链上永续合约经历了四代架构。每一代都在解决上一代的核心缺陷。Hyperliquid 是第四代——但不是终点。

---

## 1. 为什么要讲历史

理解 Hyperliquid 为什么这么设计，必须先理解它之前的尝试**为什么失败或受限**。

链上永续合约的核心矛盾从第一天就存在：

> **用户想要 CEX 的体验（快、深、便宜），但不想信任 CEX（FTX 暴雷证明了这一点）。**

每一代产品都在这个矛盾里做不同的取舍。

---

## 2. 第一代：以太坊主网上的尝试（2020-2021）

### 代表：dYdX V1-V2、Perpetual Protocol V1

**架构**：直接在以太坊主网上部署智能合约。

**dYdX V1-V2**：
- 2019 年上线，最早的链上永续之一
- 用 AMM（vAMM）做定价，不是订单簿
- 每笔交易都是以太坊 L1 交易
- Gas 费 $20-$100/笔

**Perpetual Protocol V1**：
- 2020 年上线
- 虚拟 AMM（vAMM）模式
- 部署在 xDai（后来的 Gnosis Chain）降低 Gas

**为什么失败**：

| 问题 | 原因 |
|------|------|
| Gas 太贵 | 以太坊 L1 每笔 $20-$100 |
| 太慢 | 12 秒出块，交易确认要等 |
| 流动性差 | vAMM 模式滑点大，大额交易不可能 |
| 功能受限 | 没有限价单、止损等基本功能 |

**教训**：**通用 L1 不适合做高频交易产品**。Gas 和延迟是硬伤。

---

## 3. 第二代：L2 + 链下撮合（2021-2023）

### 代表：dYdX V3（StarkEx）

**架构**：链下订单簿撮合 + StarkEx（ZK-Rollup）链上结算。

**dYdX V3 的设计**：
- 订单簿和撮合引擎**完全在链下**（由 dYdX 公司运营）
- 用户资金锁在 StarkEx 的 L1 合约里
- 每隔一段时间，StarkEx 把一批交易的 ZK 证明提交到以太坊
- 用户体验接近 CEX（毫秒级撮合）

**为什么成功（一段时间）**：
- 2021-2023 年链上永续交易量第一
- 峰值日交易量 $10B+
- 体验确实接近 CEX

**为什么最终被超越**：

| 问题 | 原因 |
|------|------|
| **中心化撮合** | 订单簿由 dYdX 公司运营，和 CEX 没本质区别 |
| **审查风险** | dYdX 可以拒绝你的订单、冻结你的账户 |
| **StarkEx 限制** | 吞吐有上限，不能无限扩展 |
| **不可组合** | 其他 DeFi 协议无法和 dYdX 的订单簿交互 |
| **Token 激励依赖** | 大量交易量靠 DYDX Token 挖矿驱动，不是真实需求 |

**dYdX V3 的核心矛盾**：它解决了性能问题，但**牺牲了去中心化**。本质上是"一个有 ZK 证明的中心化交易所"。

### dYdX V4：迁移到 Cosmos

2022 年 dYdX 宣布迁移到 Cosmos（自建应用链）。2023 年 10 月 dYdX Chain 上线。

**V4 的改进**：
- 订单簿撮合由**验证者网络**运行（不再是 dYdX 公司单独运营）
- 完全去中心化的订单簿（链下内存撮合 + 链上状态确认）
- 自己的 Cosmos SDK 链，不依赖以太坊

**V4 的问题**：
- 迁移成本巨大（用户、流动性、生态都要重建）
- Cosmos 生态相对孤立（和 EVM 世界不互通）
- 性能仍不如 Hyperliquid（Cosmos 的 Tendermint 共识有吞吐上限）
- 市场份额从 #1 跌到 #2（被 Hyperliquid 超越）

**教训**：**自建链是对的方向，但迁移的执行风险巨大**。Hyperliquid 从 Day 1 就是自建链，没有迁移包袱。

---

## 4. 第三代：AMM 模式的巅峰（2021-2024）

### 代表：GMX（Arbitrum）、Gains Network

**架构**：不用订单簿，用**流动性池 + Oracle 定价**。

**GMX 的设计**：
- 用户交易的对手方是 **GLP 池**（一个多资产流动性池）
- 价格由 **Chainlink Oracle** 提供（不是市场撮合）
- LP 存入资产到 GLP → 赚取交易者的亏损 + 手续费
- 交易者赚的钱 = LP 亏的钱（零和博弈）

**为什么 GMX 火了**：
- 2022-2023 年 Arbitrum 生态爆发
- GLP 年化 20-40%（牛市时）
- 无滑点（Oracle 定价，不管交易多大都是同一个价格）
- 简单易懂（不需要理解订单簿）

**GMX 的根本限制**：

| 问题 | 原因 |
|------|------|
| **OI 上限** | GLP 池大小决定了最大未平仓合约，不能无限扩展 |
| **LP 风险** | 交易者如果集体赚钱，LP 就亏钱（2022 年 Luna 崩盘时 GLP 大亏） |
| **无价格发现** | 价格来自 Oracle，不是市场。GMX 不能"发现"价格，只能"跟随"价格 |
| **无限价单** | Oracle 模式下限价单实现困难 |
| **可操纵** | 大户可以在 CEX 操纵价格 → Oracle 跟随 → 在 GMX 上获利 |

**GMX V2 的改进**：
- 引入隔离池（每个交易对独立池子）
- 加入资金费率
- 但核心模式没变：仍然是 Oracle + LP 池

**教训**：**AMM/Oracle 模式适合中小规模，但无法替代订单簿做大规模价格发现**。GMX 的天花板是 GLP 池的大小。

---

## 5. 第四代：应用链 + 全链上订单簿（2023-至今）

### 代表：Hyperliquid

**架构**：自建 L1 + 全链上 CLOB + 确定性执行。

**Hyperliquid 的核心突破**：

| 维度 | 之前的最优解 | Hyperliquid |
|------|------------|-------------|
| 撮合 | 链下（dYdX V3）或 Oracle（GMX） | **全链上 CLOB** |
| 性能 | 受通用链限制 | **自建链，200K TPS** |
| 最终性 | 秒级（Cosmos）或分钟级（ZK） | **0.2 秒** |
| 去中心化 | 部分（dYdX V3 中心化撮合） | **验证者网络** |
| 可组合性 | 无（dYdX V3）或有限（GMX） | **HyperEVM 原子组合** |
| 资金效率 | 低（GMX 需要大量 LP） | **高（订单簿，做市商提供流动性）** |

**为什么 Hyperliquid 能做到**：

1. **HyperBFT 共识**：Hotstuff 变体，专门为高吞吐低延迟优化
2. **确定性执行**：每个订单的执行结果在提交时就确定，不存在"交易被 revert"
3. **Gas-free 订单**：下单/撤单不消耗 Gas（只有成交时才有链上状态变化）
4. **从 Day 1 就是自建链**：没有迁移包袱，架构从头为交易优化

## 6. 四代架构对比总表

| 维度 | 第一代（ETH L1） | 第二代（L2+链下） | 第三代（AMM+Oracle） | 第四代（应用链+CLOB） |
|------|-----------------|------------------|--------------------|--------------------|
| 代表 | dYdX V1, Perp V1 | dYdX V3 (StarkEx) | GMX, Gains | **Hyperliquid** |
| 时间 | 2020-2021 | 2021-2023 | 2021-2024 | 2023-至今 |
| 撮合方式 | vAMM | 链下 CLOB | Oracle + LP 池 | **链上 CLOB** |
| 性能 | 极差（12s 出块） | 好（链下撮合） | 中（受 L2 限制） | **极好（200K TPS）** |
| 去中心化 | 高（但没人用） | 低（中心化撮合） | 中（Oracle 依赖） | **中高（验证者网络）** |
| 价格发现 | 有（但流动性差） | 有 | **无**（跟随 Oracle） | **有** |
| 可组合性 | 有（EVM） | 无 | 有（EVM） | **有（HyperEVM）** |
| 资金效率 | 低 | 高 | 低（需要大量 LP） | **高** |
| 用户体验 | 差 | 好 | 中 | **好** |
| 市场份额（峰值） | <5% | ~50% | ~30% | **~80%** |

## 7. 为什么"自建链"成为趋势

Hyperliquid 不是唯一走"自建链"路线的：

| 项目 | 链 | 用途 |
|------|-----|------|
| Hyperliquid | HyperBFT L1 | 永续合约 |
| dYdX V4 | Cosmos SDK | 永续合约 |
| Sei | Cosmos SDK | 通用交易链 |
| Injective | Cosmos SDK | 衍生品 |

**为什么自建链**：

1. **性能可控**：不受通用链的 Gas 竞争和拥堵影响
2. **费用模型自定义**：可以让下单免费、只对成交收费
3. **确定性执行**：消除"交易被 MEV 攻击"的可能
4. **垂直整合**：从共识到执行到结算，全栈优化

**代价**：
- 安全性依赖自己的验证者网络（不如以太坊的数千个节点）
- 生态从零开始（没有以太坊的 DeFi 乐高）
- 桥接风险（资金从其他链桥过来有安全隐患）

## 9. 对下一代的启示

Hyperliquid 是第四代，但不是终点。可能的第五代方向：

**1. 完全去中心化的排序器**

Hyperliquid 目前只有 16 个验证者，且排序器（sequencer）仍有中心化成分。未来可能需要更去中心化的排序方案。

**2. 跨链统一流动性**

目前 Hyperliquid 的流动性和以太坊 / Solana 是隔离的。未来可能出现"跨链永续聚合器"，把多个链上永续的流动性统一。

**3. ZK 证明 + 应用链**

结合 ZK-Rollup 的安全性和应用链的性能。目前还没有成熟方案，但 StarkNet 和 zkSync 在探索。

**4. AI 驱动的做市和风控**

做市策略、清算触发、风险参数调整由 AI 实时优化。已经有团队在 Hyperliquid 上跑 AI 做市 Vault。

## 经验教训

1. **每一代都在解决上一代的核心缺陷** —— 但也引入新的 tradeoff
2. **"自建链"是当前的最优解** —— 但代价是安全性和生态从零开始
3. **订单簿 > AMM（对大规模交易）** —— GMX 证明了 AMM 有天花板
4. **FTX 暴雷是链上永续的催化剂** —— 用户不再信任 CEX，但仍需要 CEX 级体验
5. **Token 激励不能替代真实需求** —— dYdX V3 的交易量很多是挖矿驱动，Hyperliquid 没有挖矿但交易量更大
6. **迁移是最大的执行风险** —— dYdX V3→V4 的迁移让它丢了 #1 位置
7. **从 Day 1 就做对架构选择** —— Hyperliquid 没有历史包袱，这是它最大的优势
